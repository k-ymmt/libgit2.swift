# Git2 Swift Wrapper — v0.4a Write Foundation

**Status:** Draft
**Date:** 2026-04-20
**Scope:** First slice of the v0.4 write-operations line. Adds the minimum ODB-write surface needed to create commits end-to-end from public API: blob creation, flat tree construction, commit creation, branch create/delete, annotated + lightweight tag create/delete, and a generic `Reference.delete()`. Builds on the v0.2 foundation (`docs/superpowers/specs/2026-04-20-git2-swift-wrapper-foundation-design.md`) and v0.3 read extensions (`docs/superpowers/specs/2026-04-20-git2-v0.3-read-extensions-design.md`).

## 1. Purpose

`Git2` v0.2 / v0.3 cover the full read-only surface. Any caller that needs to actually modify history has to drop down to `@_exported import Cgit2` and wrangle `git_treebuilder_*` / `git_commit_create` / `git_branch_create` / `git_tag_create` by hand. The Git2 test fixture (`Tests/Git2Tests/Support/TestFixture.swift`) is itself a several-hundred-line demonstration of how much boilerplate that requires.

v0.4a removes that hole. It adds the smallest set of write APIs that can express:

- creating a blob from in-memory `Data`,
- constructing a new tree from a flat list of entries,
- creating a commit (initial, regular, or merge) that optionally advances a ref atomically,
- creating and deleting local branches,
- creating lightweight and annotated tags, and deleting tags by name,
- deleting a generic reference.

That is enough to re-implement the current `TestFixture` helpers without `Cgit2`, and enough for user code to build history programmatically.

Index-based staging, working-directory checkout, and HEAD manipulation remain out of scope — they are the v0.4b slice.

v0.4a is **additive**: every existing v0.2 / v0.3 public API keeps the same signature and behavior.

## 2. Target Platforms

Unchanged from v0.3: Apple platforms only, macOS 13+, iOS 16+, Swift 6.2.

## 3. Scope

### 3.1 Included in v0.4a

| Area | Additions |
|---|---|
| Values | `TreeBuilderEntry` |
| Repository | `createBlob(data:)`, `tree(entries:)`, `commit(tree:parents:author:committer:message:updatingRef:)`, `createBranch(named:at:force:)`, `deleteBranch(named:)`, `createLightweightTag(named:target:force:)`, `createAnnotatedTag(named:target:tagger:message:force:)`, `deleteTag(named:)` |
| Reference | `delete()` |
| Tree | Internal `TreeEntry.FileMode.raw: git_filemode_t` helper (round-trip for the builder) |

### 3.2 Explicitly not included in v0.4a

- **Index / staging.** `git_index_*` wrappers (read, write, add, remove, conflict enumeration).
- **Checkout / working-directory manipulation.** `git_checkout_*`.
- **HEAD set / branch switching.** `git_repository_set_head`, `git_repository_set_head_detached`.
- **Stateful `TreeBuilder` class.** Incremental insert/remove and derivation from an existing tree. Deferred until a concrete use case appears.
- **Blob from file / stream.** `createBlob(fromFileAt: URL)`, streaming variants.
- **Tags of non-commits.** `target: Object` overloads for `createLightweightTag` / `createAnnotatedTag` (tag-of-tag, tag-of-tree).
- **Remote branches.** `GIT_BRANCH_REMOTE` creation / deletion.
- **Signed commits.** `git_commit_create_with_signature`.
- **`message_encoding` parameter on `commit(...)`**. libgit2 is passed `nil` (treat as UTF-8).
- **Reflog inspection / manipulation** outside whatever `git_branch_delete` cleans up automatically.
- **`Tag.delete()` or `Branch.delete()` instance methods.** Deletion is always name-based via `Repository`.

### 3.3 Non-goals (unchanged from v0.3)

- No shipped CLI.
- No 1:1 exposure of every libgit2 function; low-level access remains via `@_exported import Cgit2`.
- No Linux / Windows / Android support.
- No Keychain / credential-UI integration.
- No SHA-256 repositories.

## 4. Architectural Continuity

Every v0.2 / v0.3 decision carries forward unchanged:

- **Hybrid resource ownership.** New write paths return the same handle types as v0.3 (`Tree`, `Commit`, `Tag`, `Reference`) or snapshot values (`OID`). No new handle class is introduced.
- **Parent-child lifetime.** Handles returned from write APIs hold a strong reference to their owning `Repository`. `TreeBuilderEntry` is a snapshot value and has no such tie.
- **Typed throws.** Every write API uses `throws(GitError)`.
- **Per-`Repository` serialization.** All libgit2 calls — including the new write paths — run inside `repository.lock.withLock`. Multi-step operations (e.g. `git_treebuilder_new` → insert loop → `git_treebuilder_write` → `git_tree_lookup`) are held inside a single critical section.
- **Error representation.** Still `GitError` with `Code` / `Class` enums; no new error types.
- **Lifecycle.** Still `Git.bootstrap()` / `Git.shutdown()`.

## 6. Types

### 6.1 `TreeBuilderEntry`

```swift
// Sources/Git2/Values/TreeBuilderEntry.swift

/// One entry to insert when building a new tree via
/// ``Repository/tree(entries:)``.
public struct TreeBuilderEntry: Sendable, Equatable {
    /// The entry's basename (no path separators).
    public let name: String

    /// The OID of the blob, sub-tree, or submodule commit the entry points at.
    public let oid: OID

    /// The Git filemode — distinguishes regular blob, executable blob,
    /// symbolic link, sub-tree, and submodule.
    public let filemode: TreeEntry.FileMode

    public init(name: String, oid: OID, filemode: TreeEntry.FileMode)
}
```

Reusing `TreeEntry.FileMode` (already public in v0.3) keeps the filemode vocabulary the same for read and write.

### 6.2 `TreeEntry.FileMode.raw` (internal)

Round-trip helper added to `Sources/Git2/Objects/Tree.swift`:

```swift
extension TreeEntry.FileMode {
    internal var raw: git_filemode_t {
        switch self {
        case .tree:            return GIT_FILEMODE_TREE
        case .blob:            return GIT_FILEMODE_BLOB
        case .blobExecutable:  return GIT_FILEMODE_BLOB_EXECUTABLE
        case .link:            return GIT_FILEMODE_LINK
        case .commit:          return GIT_FILEMODE_COMMIT
        }
    }
}
```

## 7. Public API

### 7.1 `Repository.createBlob(data:)`

```swift
extension Repository {
    /// Writes `data` to the object database as a blob and returns its OID.
    ///
    /// The blob becomes part of the ODB immediately. Nothing else references it
    /// until a tree containing this OID is written.
    public func createBlob(data: Data) throws(GitError) -> OID
}
```

**Return type.** `OID` (not `Blob`). The primary use is "feed this OID into a `TreeBuilderEntry`." Callers that want a `Blob` handle can re-hydrate via `try repository.object(for: oid)`.

**Empty `Data`.** Accepted. Produces the well-known empty-blob OID `e69de29bb2d1d6434b8b29ae775ad8c2e48c5391`. The implementation passes a non-nil dummy pointer when `data.count == 0` (matches the defensive path in the existing test fixture).

### 7.2 `Repository.tree(entries:)`

```swift
extension Repository {
    /// Builds a new tree from a flat list of entries and writes it to the ODB.
    ///
    /// All entries must be top-level of the new tree. Sub-directories are
    /// represented by entries whose ``TreeBuilderEntry/filemode`` is `.tree`
    /// and whose ``TreeBuilderEntry/oid`` is an existing tree OID.
    ///
    /// - Throws: ``GitError`` — including ``GitError/Code/exists`` if two entries
    ///   share a name, and ``GitError/Code/invalid`` if a filemode does not
    ///   match the kind of the referenced object.
    public func tree(entries: [TreeBuilderEntry]) throws(GitError) -> Tree
}
```

**Ordering.** libgit2 sorts entries internally; the caller does not have to pre-sort.

**Empty `entries`.** Accepted. Produces the well-known empty-tree OID `4b825dc642cb6eb9a060e54bf8d69288fbee4904`.

**Nested trees.** Built bottom-up: the caller builds child trees first, then references their OIDs from a parent `TreeBuilderEntry` with `.tree` filemode. v0.4a does not ship a recursive helper.

**Lifetime.** The treebuilder handle is allocated and freed inside a single `lock.withLock` call. If any intermediate step throws, `git_treebuilder_free` in a `defer` cleans up; nothing has been written to the ODB until `git_treebuilder_write` succeeds.

### 7.3 `Repository.commit(tree:parents:...)`

```swift
extension Repository {
    /// Writes a new commit to the ODB and optionally advances a reference to it.
    ///
    /// - Parameters:
    ///   - tree: The tree that represents the commit's file snapshot.
    ///   - parents: The parent commits in the order Git should record them.
    ///     Empty for an initial commit, one for a regular commit, two or more
    ///     for a merge commit.
    ///   - author: The author signature.
    ///   - committer: The committer signature. Defaults to `author`.
    ///   - message: The commit message. libgit2 does not append a trailing
    ///     newline — the caller should include one if desired.
    ///   - updatingRef: If non-nil, the reference to advance atomically to the
    ///     new commit. Examples: `"HEAD"`, `"refs/heads/main"`. The ref need
    ///     not exist yet (supports initial commit on an unborn HEAD). If nil,
    ///     the commit is created but no reference points at it.
    /// - Returns: The created ``Commit`` handle.
    public func commit(
        tree: Tree,
        parents: [Commit],
        author: Signature,
        committer: Signature? = nil,
        message: String,
        updatingRef: String? = nil
    ) throws(GitError) -> Commit
}
```

**Committer default.** `committer ?? author`. Matches the most common case (`git commit -m`) and keeps call sites short.

**Message encoding.** libgit2 is passed `nil`, which it treats as UTF-8. No public `encoding` parameter in v0.4a.

**Atomicity.** When `updatingRef` is non-nil, libgit2's `git_commit_create` writes the commit object and advances the ref as a single logical step; the caller does not have to roll back on partial failure.

**Cross-repo handles.** `tree` and each `parents[i]` are assumed to belong to the same `Repository` as `self`. v0.4a does not validate this at the Swift layer; libgit2 surfaces a runtime error (typically `.odb` class) if the ODB does not contain the referenced OIDs.

### 7.4 `Repository.createBranch` / `Repository.deleteBranch`

```swift
extension Repository {
    /// Creates a local branch `refs/heads/<name>` pointing at `target`.
    public func createBranch(
        named name: String,
        at target: Commit,
        force: Bool = false
    ) throws(GitError) -> Reference

    /// Deletes the local branch `refs/heads/<name>`.
    ///
    /// libgit2 refuses to delete the branch currently checked out through HEAD.
    public func deleteBranch(named name: String) throws(GitError)
}
```

**Name form.** Short name only (`"feature"`, not `"refs/heads/feature"`). libgit2's `git_branch_create` / `git_branch_lookup` require the short form.

**Remote branches.** Not supported in v0.4a (always `GIT_BRANCH_LOCAL`). A future `createRemoteBranch` / `deleteRemoteBranch` is an additive follow-up.

**Delete implementation.** `git_branch_lookup` → `git_branch_delete`, both under a single lock. `git_branch_delete` cleans up the branch reflog and tracking-config entries — meaningfully more than `git_reference_delete`.

### 7.5 `Repository.createLightweightTag` / `createAnnotatedTag` / `deleteTag`

```swift
extension Repository {
    /// Creates a lightweight tag — a plain `refs/tags/<name>` that points
    /// directly at `target`. No tag object is written.
    public func createLightweightTag(
        named name: String,
        target: Commit,
        force: Bool = false
    ) throws(GitError) -> Reference

    /// Creates an annotated tag — writes a `tag` object carrying `tagger` and
    /// `message`, and installs `refs/tags/<name>` pointing at that object.
    public func createAnnotatedTag(
        named name: String,
        target: Commit,
        tagger: Signature,
        message: String,
        force: Bool = false
    ) throws(GitError) -> Tag

    /// Deletes the tag `refs/tags/<name>`.
    public func deleteTag(named name: String) throws(GitError)
}
```

**Name form.** Short name only (`"v1.0"`, not `"refs/tags/v1.0"`). Matches `git_tag_create*` / `git_tag_delete` contract.

**Target kind.** v0.4a restricts `target` to `Commit`. `git_tag_create*` internally takes a `git_object *`; the Swift `Commit.handle` (`git_commit *`) is passed directly because the `git_*` family is a subtype hierarchy and `OpaquePointer` is already type-erased. Tag-of-tag / tag-of-tree is deferred to a future `Object`-accepting overload.

**Lightweight tag return.** `git_tag_create_lightweight` returns only the target OID; the implementation follows up with `git_reference_lookup("refs/tags/\(name)")` inside the same critical section so the returned value has the same shape as `createBranch`.

**Annotated tag return.** `Tag` handle, re-hydrated via `git_tag_lookup` immediately after `git_tag_create` so the caller can inspect `tag.name` / `tag.message` / `tag.tagger` / `tag.targetOID` without a second round trip.

**Delete scope.** `git_tag_delete` removes `refs/tags/<name>` for both lightweight and annotated tags. The annotated tag's tag-object becomes unreferenced in the ODB but is not removed (standard Git behavior — it will be pruned by GC).

### 7.6 `Reference.delete()`

```swift
extension Reference {
    /// Deletes this reference from the repository.
    ///
    /// Uses `git_reference_delete`, which does not touch reflog or config.
    /// For branches, prefer ``Repository/deleteBranch(named:)`` so reflog and
    /// tracking config are cleaned up too.
    ///
    /// After deletion, this ``Reference`` handle still exists in memory but
    /// further operations on it will throw or return stale data.
    public func delete() throws(GitError)
}
```

`git_reference_delete` is the right tool for deleting refs that are neither branches nor tags (e.g., `refs/notes/*`, caller-invented refs). Branches and tags have their reflog/config-aware deletes on `Repository`.

## 8. Internal Infrastructure

### 8.1 `signatureHandle(for:)`

`git_signature *` allocation is needed twice: commit creation and annotated tag creation. A single helper avoids duplicating the conversion:

```swift
// Sources/Git2/Repository/Repository+Commits.swift
extension Repository {
    internal func signatureHandle(for sig: Signature)
        throws(GitError) -> UnsafeMutablePointer<git_signature>
    {
        var out: UnsafeMutablePointer<git_signature>?
        try check(
            git_signature_new(
                &out,
                sig.name, sig.email,
                git_time_t(sig.date.timeIntervalSince1970),
                Int32(sig.timeZone.secondsFromGMT() / 60)
            )
        )
        return out!
    }
}
```

The caller owns the result and is responsible for `git_signature_free`. All callers hold `repository.lock` when invoking this helper — the function itself does not acquire the lock.

### 8.2 Lock discipline

Every write API takes `repository.lock.withLock` once at the top and performs all libgit2 calls — including multi-step flows like `treebuilder_new` → insert* → `treebuilder_write` → `tree_lookup` — inside that single call. This mirrors the discipline v0.2 / v0.3 established for read paths and keeps partial state invisible to concurrent readers.

### 9.2 Error-code specificity

Every failure path that maps to a specific `GitError.Code` is asserted by code (e.g. `.exists`, `.notFound`), not just by throw-type. This matches the tightening v0.3 followed up on for `invalidSpec` in reference lookup.

### 9.3 `TestFixture` rewrite

`Tests/Git2Tests/Support/TestFixture.swift` replaces its libgit2 direct calls with the new public API. The function signatures stay stable so existing callers (Log / Diff / RevWalk / Reference / Concurrency test suites) continue to compile unchanged.

Indicative before → after for `makeLinearHistory`:

```swift
// Before
var blobID = git_oid()
try entry.message.withCString { bytes in
    let r = git_blob_create_from_buffer(&blobID, repo, UnsafeRawPointer(bytes), strlen(bytes))
    guard r == 0 else { throw GitError.fromLibgit2(r) }
}
var builder: OpaquePointer?
// ... treebuilder allocation, insert, write, lookup, signature_new ...
let r: Int32 = entry.message.withCString { msg in
    git_commit_create(&commitID, repo, "HEAD", signature, signature, "UTF-8", msg, treeH, 1, buf.baseAddress)
}

// After
let blobOID = try repo.createBlob(data: Data(entry.message.utf8))
let tree = try repo.tree(entries: [
    .init(name: "README.md", oid: blobOID, filemode: .blob)
])
let commit = try repo.commit(
    tree: tree,
    parents: parents,
    author: entry.author,
    message: entry.message,
    updatingRef: "HEAD"
)
```

The `TreeEntryDescription` struct on `TestFixture` (used by `makeCommitWithTree`) currently holds a `git_filemode_t`. Migrating it to `TreeEntry.FileMode` is desirable for a full `Cgit2`-import removal but is **not** part of v0.4a — it is a mechanical follow-up that would balloon the diff with call-site churn. It lands on TODO.md as a deferred polish.

The `makeAnnotatedTag` / `makeBranches` helpers are rewritten onto `createAnnotatedTag` / `createBranch` directly, removing the `git_repository_open` / signature-allocation boilerplate those helpers carry today.

### 9.4 Regression coverage

Because `TestFixture` is the backbone of every existing test suite, rewriting it onto v0.4a public API means the full existing test run (Log, Diff, RevWalk, Reference, Concurrency, Object, Tree, Blob, Tag) retroactively verifies that the new write APIs produce objects indistinguishable from the libgit2-direct path. This is a deliberate design property — the v0.4a rewrite moves TestFixture from a "trust libgit2" fixture to a "trust our API" fixture.

### 9.5 Known test-coverage gaps (accepted)

- **`GIT_FILEMODE_UNREADABLE` round-trip.** The internal `TreeEntry.FileMode.raw` does not emit this sentinel (there is no public case for it). Trying to build a tree with it is impossible by construction. No test needed.
- **Cross-repository misuse.** Passing a `Tree` / `Commit` / `Signature` belonging to a different `Repository` is not type-prevented. libgit2 surfaces an error at write time; v0.4a does not add a Swift-layer assertion.
- **Very large blobs / wide trees.** Performance behavior is not measured. Functional correctness is covered by smoke tests only.

## 10. Roadmap

### 10.1 Immediate follow-up (v0.4b)

- **Index / staging.** `git_index_*` wrapper: read, add, remove, write, conflict enumeration.
- **Checkout.** `git_checkout_head`, `git_checkout_tree`, `git_checkout_index`. Working-directory mutation with safety options.
- **HEAD manipulation.** `git_repository_set_head`, `git_repository_set_head_detached`, branch switching.

### 10.2 Deferred from v0.4a (additive, non-breaking)

- **Stateful `TreeBuilder`.** Public handle with `insert(name:oid:filemode:)`, `remove(name:)`, `derived(from: Tree)`, `write() -> Tree`. Needed when a caller wants to edit an existing tree without reconstructing every entry.
- **`createBlob(fromFileAt: URL)` and streaming variants.** For large files or files already on disk.
- **Tag of non-commit.** `target: Object` overloads on `createLightweightTag` / `createAnnotatedTag`.
- **Remote branches.** `createRemoteBranch` / `deleteRemoteBranch` wrapping `GIT_BRANCH_REMOTE`.
- **Signed commits.** `commit(tree:parents:..., signature: String)` mapping onto `git_commit_create_with_signature`.
- **`message_encoding:` parameter** on `commit(...)`.
- **TestFixture `TreeEntryDescription` type migration.** Replace `git_filemode_t` with `TreeEntry.FileMode` and drop the last `@testable` use of `Cgit2` in the fixture layer.

### 10.3 Versioning

v0.4a ships as a minor bump (e.g. v0.4.0) since it is fully additive. v0.4b, once landed, is another minor bump (v0.4.1 or v0.5.0 depending on how deep its changes go). The "a / b" suffixes in this design document are for tracking only — they are not intended as release tags.
