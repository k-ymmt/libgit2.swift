# Git2 Swift Wrapper — v0.5b-i Network (Fetch) Slice

**Status:** Draft
**Date:** 2026-04-21
**Scope:** First slice of the v0.5b "network" line. Adds the `git_remote_*`
family's read / CRUD / fetch surface, the `git_remote_callbacks` bridge for
Swift closures, and the supporting value types (`Credential`,
`CertificateVerdict`, `TransferProgress`, `Repository.FetchOptions`,
`Refspec`). Builds on v0.2
(`docs/superpowers/specs/2026-04-20-git2-swift-wrapper-foundation-design.md`),
v0.3 (`docs/superpowers/specs/2026-04-20-git2-v0.3-read-extensions-design.md`),
v0.4a
(`docs/superpowers/specs/2026-04-20-git2-v0.4a-write-foundation-design.md`),
v0.4b-i
(`docs/superpowers/specs/2026-04-20-git2-v0.4b-i-index-design.md`), v0.4b-ii
(`docs/superpowers/specs/2026-04-20-git2-v0.4b-ii-checkout-head-design.md`),
v0.5a-i
(`docs/superpowers/specs/2026-04-20-git2-v0.5a-i-merge-cherrypick-design.md`),
and v0.5a-ii
(`docs/superpowers/specs/2026-04-20-git2-v0.5a-ii-rebase-design.md`).

## 1. Purpose

v0.5a closed the local "integration operations" line (merge, cherry-pick,
rebase). What is still missing is **network I/O** — the ability to talk to a
remote Git host, resolve its refs, and download objects into the local ODB.
Without it, every workflow built on top of v0.2–v0.5a is bounded by whatever
already exists on the local disk.

v0.5b-i delivers three use cases jointly:

- **Remote management.** `createRemote`, `lookupRemote`, `remotes()`,
  `deleteRemote`, `renameRemote`, plus URL / refspec accessors and mutators.
  Everything a caller needs to hold a persistent `origin` entry in
  `.git/config`.
- **Authenticated HTTPS fetch.** `Remote.fetch(...)` wrapping
  `git_remote_fetch` with Swift closures for `credentials`,
  `certificate_check`, and `transfer_progress`. Closures are first-class
  fields on `Repository.FetchOptions`; the implementation bridges them to
  libgit2's `git_remote_callbacks` C struct.
- **Refspec-driven fetch.** `fetch(refspecs:options:)` for callers who need
  to override the remote's configured fetch refspecs (e.g. a one-off mirror
  fetch).

v0.5b-i is **additive**: every existing v0.2 / v0.3 / v0.4a / v0.4b-i /
v0.4b-ii / v0.5a-i / v0.5a-ii public API keeps the same signature and
behavior.

## 2. Target Platforms

Unchanged from v0.5a-ii: Apple platforms only, macOS 13+, iOS 16+, Swift 6.2.
HTTPS transport uses SecureTransport (selected when the v0.1 XCFramework was
built with `USE_HTTPS=SecureTransport`); no SSH (`USE_SSH=OFF`).

### 3.2 Explicitly not included in v0.5b-i

Tracked in TODO.md under "Deferred from v0.5b-i".

- **Push (v0.5b-ii).** `git_remote_push`, `PushOptions`, `push_transfer_progress`,
  `push_negotiation`, `push_update_reference`. Deliberately split so the
  callback-bridge mechanism (§4.5) can be proven on the fetch path before
  push-specific callbacks layer on top.
- **Remote primitive APIs.** `git_remote_connect`, `_connect_ext`,
  `_download`, `_update_tips`, `_upload`, `_disconnect`, `_stop`, `_ls`,
  `_connected`, `_prune`. v0.5b-i ships only the `fetch` porcelain per Q4=A;
  primitives land if a concrete decomposition use case appears.
- **Proxy options.** `git_proxy_options` with `type` / `url` /
  `credentials` / `certificate_check`. v0.5b-i ships "no proxy" (libgit2's
  default). Corporate-proxy requirements surface in a follow-up.
- **SSH credentials.** `GIT_CREDENTIAL_SSH_KEY` / `_SSH_CUSTOM` /
  `_SSH_INTERACTIVE` / `_SSH_MEMORY` cases of `git_credential_t`. `Credential`
  enum intentionally omits these; re-addable additively once libssh2 lands
  (currently out-of-scope per v0.1's `USE_SSH=OFF` build).
- **Remote callbacks beyond the three in scope.** `sideband_progress`,
  `pack_progress`, `completion`, `update_refs` / `update_tips`,
  `remote_ready`, `resolve_url`. v0.5b-i surfaces only `credentials`,
  `certificate_check`, `transfer_progress`; the rest ride the same bridge
  later.
- **Certificate detail.** `git_cert *` structure contents (X.509 DER, hostkey
  fingerprints). `CertificateVerdict` takes `host` + `isValid` only;
  inspecting certificate internals needs a `Certificate` value type that
  v0.5b-i does not design.
- **Anonymous / detached remotes.** `git_remote_create_anonymous`,
  `_create_detached`. In-memory remote use cases unverified.
- **Remote creation options.** `git_remote_create_with_opts` +
  `git_remote_create_options` (`flags`, `fetchspec`, detached repo, etc.).
  v0.5b-i takes the two simple overloads (`create(named:url:)`,
  `create(named:url:fetchspec:)`).
- **Remote instance mutation during a connection.** `_set_instance_url`,
  `_set_instance_pushurl` are used from inside the deprecated `resolve_url`
  callback or the `remote_ready` callback; both of those callbacks are
  deferred, so the instance-URL mutators are deferred with them.
- **Remote dup.** `git_remote_dup`. Duplicating a `Remote` handle has no
  demonstrated need.
- **`git_remote_oid_type`.** SHA-256 support requires
  `EXPERIMENTAL_SHA256=ON` at libgit2 build time (currently off per
  TODO.md "libgit2 build options not enabled").
- **Autotag / prune-refs config mutation.** `git_remote_set_autotag`,
  `git_remote_prune_refs`. Config-edge APIs; add when a concrete demand
  appears.
- **Task-aware cancellation.** `Task.cancel()` → libgit2 abort wiring.
  v0.5b-i offers in-closure cancellation via `transferProgress` returning
  `false`. Task-integrated cancellation requires an async bridge that
  contradicts v0.2's synchronous + lock model.
- **Shallow-clone helpers.** `depth` flows through to libgit2 but v0.5b-i
  does not add `unshallow()` / `fetch(unshallow:)` sugar.

### 3.3 Non-goals (unchanged from v0.5a-ii)

- No shipped CLI.
- No 1:1 exposure of every libgit2 function; low-level access remains via
  `@_exported import Cgit2`.
- No Linux / Windows / Android support.
- No Keychain / credential-UI integration — credentials are Swift values the
  caller supplies.
- No SHA-256 repositories.

## 4. Architectural Continuity

Every v0.2 / v0.3 / v0.4a / v0.4b-i / v0.4b-ii / v0.5a-i / v0.5a-ii decision
carries forward unchanged.

### 4.1 Handle ownership

`Remote` joins the existing pattern: a `final class`, `@unchecked Sendable`,
retaining its parent `Repository` strongly and freeing the underlying
`git_remote *` in `deinit`. Creation happens only through `Repository`
extension methods (`createRemote(...)`, `lookupRemote(...)`, or as the
transient receiver of the `fetch(remoteNamed:...)` sugar). Serialization is
the parent repository's `HandleLock`.

### 4.2 Per-`Repository` serialization

Every new method takes `repository.lock.withLock` once at the top. This
includes the `Remote` instance methods (`name`, `url`, `pushURL`,
`fetchRefspecs`, `pushRefspecs`, `defaultBranch()`, `fetch(...)`) and all
`Repository` CRUD extensions.

**Fetch holds the lock across the whole network round-trip.** The lock is
held while libgit2 opens the HTTPS connection, runs ref discovery, downloads
the packfile, updates local refs, and returns. A parallel `fetch` call on a
different remote of the same repository blocks until the first call returns.
This matches the "one-operation-at-a-time-per-Repository" contract v0.2
established; callers who want concurrent fetches open one `Repository`
handle per concurrent operation.

### 4.3 Callback re-entrancy

The three surfaced callbacks (`credentials`, `certificateCheck`,
`transferProgress`) are invoked **synchronously on the fetch thread while
the repository lock is held**. A closure that reaches back into
`repository.*` (e.g. to read a config value) will deadlock on the
`HandleLock`'s `os_unfair_lock`. The public API documents this in the
`typealias` doc comment for each handler, and the deadlock is exercised as
a negative-path unit test.

UI updates from `transferProgress` must be dispatched to the main actor
asynchronously (`Task { @MainActor in ... }`) — not done synchronously in
the closure body.

### 4.4 Annotated-commit provenance for fetched refs

v0.5a-i deferred `git_annotated_commit_from_fetchhead` because `FETCH_HEAD`
was not yet populated by any public API. v0.5b-i fills that gap: after a
successful `fetch`, `FETCH_HEAD` is written, and
`Repository.annotatedCommit(fromFetchHead:remoteURL:oid:)` wraps
`git_annotated_commit_from_fetchhead` to reconstruct an `AnnotatedCommit`
that carries fetch-origin provenance for subsequent merge / rebase calls.
This closes v0.5a-i TODO item "`git_annotated_commit_from_fetchhead`. Lands
with v0.5b remote support."

### 4.5 Swift-closure → C-callback bridge

**Problem.** `git_remote_callbacks` is four C function pointers plus
`void *payload`. Swift closures that capture state cannot be directly cast
to `@convention(c)` function pointers.

**Design.** Each `fetch` call allocates a per-call
`RemoteCallbackContext` (internal class) that owns the three closure
fields plus a slot for errors captured from throwing closures. The context
is passed to libgit2 via `cbs.payload = Unmanaged.passUnretained(ctx).toOpaque()`.
Four static `@convention(c)` trampolines read the payload back via
`Unmanaged.fromOpaque(...).takeUnretainedValue()`, dispatch to the Swift
closure, and marshal the return value back to libgit2's C ABI.

**Lifetime.** The context is a stack-local variable in `fetch(...)`; its
lifetime is extended across the `git_remote_fetch` call with
`withExtendedLifetime(ctx) { ... }`. Because libgit2 only invokes the
callbacks while `git_remote_fetch` is on the stack, the `passUnretained`
reference cannot dangle.

**Error capture.** When a Swift throwing closure throws:

1. The trampoline stores the error in `ctx.capturedError`.
2. The trampoline returns a libgit2-level error code (`-1`), which causes
   `git_remote_fetch` to unwind and return non-zero.
3. The Swift side of `fetch(...)` inspects `ctx.capturedError` before
   calling `check(result)`. If set:
   - If the captured error is already a `GitError`, it is thrown verbatim.
   - If it is any other `Error`, it is wrapped as
     `GitError(code: .user, class: .callback, message: "\(error)")` —
     typed-throws on `throws(GitError)` forbids re-throwing arbitrary
     `Error`s.

**Transfer-progress cancellation.** The `transferProgress` closure returns
`Bool` — `true` to continue, `false` to cancel. On cancel, the trampoline
returns `-1` and libgit2 aborts with `GIT_EUSER`. The Swift side surfaces
this as `GitError(code: .user, class: .callback, message: "cancelled by transfer progress")`.

### 4.6 `@Sendable` closures in a `Sendable` struct

`FetchOptions` is `Sendable`. All three closure typealiases are
`@Sendable`. This means:

- `FetchOptions` values cross actor boundaries freely.
- Closures cannot capture non-`Sendable` state (e.g. a class that tracks
  progress by mutating a property); callers must reach for `Sendable`
  state holders (`AsyncStream.Continuation`, an actor, a
  `@MainActor`-isolated closure that internally dispatches).

This is the first handler-carrying value type in the wrapper; the pattern
locks in the `@Sendable` convention for future callback-bearing options
(`PushOptions` in v0.5b-ii, the deferred callbacks slice).

### 4.7 Bare repository

`git_remote_fetch` works on bare repositories (the fetch updates
`refs/remotes/...` directly without a working tree). v0.5b-i does not add
any pre-check; libgit2's behavior is surfaced unchanged.

### 4.8 Unborn HEAD

`fetch` does not inspect or touch HEAD beyond writing `FETCH_HEAD`; an
unborn HEAD does not interfere. `annotatedCommit(fromFetchHead:remoteURL:oid:)`
likewise does not require HEAD to be born.

## 5. Public API

### 5.1 `Repository` — Remote CRUD

```swift
extension Repository {
    /// Wraps `git_remote_create`. The fetch refspec defaults to
    /// `+refs/heads/*:refs/remotes/<name>/*`.
    public func createRemote(named name: String, url: String)
        throws(GitError) -> Remote

    /// Wraps `git_remote_create_with_fetchspec`.
    public func createRemote(named name: String, url: String, fetchspec: String)
        throws(GitError) -> Remote

    /// Wraps `git_remote_lookup`. Throws `.notFound` (`class: .config`) if the
    /// remote is not configured.
    public func lookupRemote(named name: String) throws(GitError) -> Remote

    /// Wraps `git_remote_list`. Order matches libgit2's return (typically
    /// insertion order in `.git/config`).
    public func remotes() throws(GitError) -> [String]

    /// Wraps `git_remote_delete`. Removes the `[remote "<name>"]` block and
    /// all remote-tracking refs under `refs/remotes/<name>/`.
    public func deleteRemote(named name: String) throws(GitError)

    /// Wraps `git_remote_rename`. Returns any refspecs libgit2 could not
    /// rewrite (e.g. refspecs that do not mention the old name). The typical
    /// success case is an empty array.
    @discardableResult
    public func renameRemote(from oldName: String, to newName: String)
        throws(GitError) -> [String]

    /// Wraps `git_remote_name_is_valid`. Pure validation; does not touch
    /// config.
    public static func isValidRemoteName(_ name: String) -> Bool

    /// Wraps `git_remote_set_url`.
    public func setRemoteURL(named name: String, to url: String)
        throws(GitError)

    /// Wraps `git_remote_set_pushurl`. Pass `nil` to clear (push falls back
    /// to the fetch URL).
    public func setRemotePushURL(named name: String, to url: String?)
        throws(GitError)

    /// Wraps `git_remote_add_fetch`.
    public func addFetchRefspec(remoteNamed name: String, refspec: Refspec)
        throws(GitError)

    /// Wraps `git_remote_add_push`.
    public func addPushRefspec(remoteNamed name: String, refspec: Refspec)
        throws(GitError)
}
```

### 5.3 `Remote` handle

```swift
public final class Remote: @unchecked Sendable {
    public let repository: Repository

    /// Wraps `git_remote_name`. `nil` only for in-memory / anonymous
    /// remotes, which v0.5b-i does not produce — kept nullable to match
    /// libgit2.
    public var name: String? { get }

    /// Wraps `git_remote_url`.
    public var url: String? { get }

    /// Wraps `git_remote_pushurl`. `nil` means "push uses the same URL as
    /// fetch".
    public var pushURL: String? { get }

    /// Wraps `git_remote_get_fetch_refspecs`.
    public var fetchRefspecs: [Refspec] { get throws(GitError) }

    /// Wraps `git_remote_get_push_refspecs`.
    public var pushRefspecs: [Refspec] { get throws(GitError) }

    /// Wraps `git_remote_default_branch`. Requires a connection established
    /// by a prior `fetch(...)`; throws `.unbornBranch` or `.notFound` on
    /// unset / unavailable.
    public func defaultBranch() throws(GitError) -> String

    /// Wraps `git_remote_fetch`. Pass `nil` for `refspecs` to use the
    /// remote's configured fetch refspecs.
    public func fetch(
        refspecs: [Refspec]? = nil,
        options: Repository.FetchOptions = Repository.FetchOptions(),
        reflogMessage: String? = nil
    ) throws(GitError)
}
```

### 5.4 Value types

```swift
public enum Credential: Sendable {
    case userPass(username: String, password: String)
    case `default`
    case username(String)

    public struct AllowedTypes: OptionSet, Sendable {
        public let rawValue: UInt32
        public init(rawValue: UInt32)
        public static let userpassPlaintext: AllowedTypes  // 1 << 0
        public static let `default`:         AllowedTypes  // 1 << 3
        public static let username:          AllowedTypes  // 1 << 5
    }
}

public enum CertificateVerdict: Sendable {
    case accept        // → 0. Allow the connection to proceed.
    case reject        // → -1. Abort; `fetch` throws.
    case passthrough   // → GIT_PASSTHROUGH. Fall back to libgit2 / SecureTransport
                       //   default validation (i.e. behave as if no custom handler
                       //   were set).
}

public struct TransferProgress: Sendable {
    public let totalObjects:    Int
    public let indexedObjects:  Int
    public let receivedObjects: Int
    public let localObjects:    Int
    public let totalDeltas:     Int
    public let indexedDeltas:   Int
    public let receivedBytes:   Int
    public var fractionCompleted: Double { get }  // indexedObjects / max(totalObjects,1)
}

public struct Refspec: Sendable, Hashable {
    public let string: String
    public init(_ string: String)
}

extension Repository {
    public struct FetchOptions: Sendable {
        public typealias CredentialsHandler =
            @Sendable (_ url: String, _ usernameFromURL: String?, _ allowed: Credential.AllowedTypes)
                throws -> Credential
        public typealias CertificateCheckHandler =
            @Sendable (_ host: String, _ isValid: Bool) -> CertificateVerdict
        public typealias TransferProgressHandler =
            @Sendable (TransferProgress) -> Bool   // false → cancel

        public var credentials:       CredentialsHandler?
        public var certificateCheck:  CertificateCheckHandler?
        public var transferProgress:  TransferProgressHandler?

        public var prune:           PruneSetting    = .unspecified
        public var updateFetchHead: Bool            = true
        public var downloadTags:    AutotagOption   = .unspecified
        public var depth:           Int             = 0              // 0 = full
        public var followRedirects: RedirectPolicy  = .initial
        public var customHeaders:   [String]        = []

        public init()

        public enum PruneSetting: Sendable { case unspecified, prune, noPrune }
        public enum AutotagOption: Sendable { case unspecified, auto, none, all }
        public enum RedirectPolicy: Sendable { case none, initial, all }
    }
}
```

## 6. Callback Bridge Implementation Sketch

```swift
// Sources/Git2/Internal/RemoteCallbackBridge.swift

internal final class RemoteCallbackContext {
    let credentials:       Repository.FetchOptions.CredentialsHandler?
    let certificateCheck:  Repository.FetchOptions.CertificateCheckHandler?
    let transferProgress:  Repository.FetchOptions.TransferProgressHandler?
    var capturedError:     (any Error)?

    init(_ options: Repository.FetchOptions) {
        self.credentials      = options.credentials
        self.certificateCheck = options.certificateCheck
        self.transferProgress = options.transferProgress
    }
}

internal func withRemoteCallbacks<R>(
    _ options: Repository.FetchOptions,
    body: (UnsafeMutablePointer<git_remote_callbacks>, RemoteCallbackContext) -> R
) -> R {
    var cbs = git_remote_callbacks()
    git_remote_init_callbacks(&cbs, UInt32(GIT_REMOTE_CALLBACKS_VERSION))
    let ctx = RemoteCallbackContext(options)
    cbs.payload = Unmanaged.passUnretained(ctx).toOpaque()
    if options.credentials       != nil { cbs.credentials       = remoteBridge_credentials }
    if options.certificateCheck  != nil { cbs.certificate_check = remoteBridge_certCheck }
    if options.transferProgress  != nil { cbs.transfer_progress = remoteBridge_transfer }
    return withExtendedLifetime(ctx) { body(&cbs, ctx) }
}

private func remoteBridge_credentials(
    out: UnsafeMutablePointer<OpaquePointer?>?,
    url: UnsafePointer<CChar>?,
    usernameFromURL: UnsafePointer<CChar>?,
    allowedTypes: UInt32,
    payload: UnsafeMutableRawPointer?
) -> Int32 {
    guard let payload, let out else { return -1 }
    let ctx = Unmanaged<RemoteCallbackContext>.fromOpaque(payload).takeUnretainedValue()
    guard let handler = ctx.credentials else { return Int32(GIT_PASSTHROUGH.rawValue) }
    do {
        let cred = try handler(
            url.map(String.init(cString:)) ?? "",
            usernameFromURL.map(String.init(cString:)),
            Credential.AllowedTypes(rawValue: allowedTypes)
        )
        return cred.createGitCredential(out: out)   // -> git_credential_userpass_plaintext_new etc.
    } catch {
        ctx.capturedError = error
        return -1
    }
}

// remoteBridge_certCheck / remoteBridge_transfer similar, same dispatch pattern
```

Each trampoline is a free function (no captured state, eligible for
`@convention(c)` conversion).

```bash
#!/usr/bin/env bash
set -euo pipefail
```

Test cases exercised against the live repo:

1. **Successful authenticated fetch.** `credentials` closure returns
   `.userPass(username: "x-access-token", password: token)`. Assert `fetch`
   returns, `refs/remotes/origin/main` points to a non-zero OID, and
   `credentials` / `certificateCheck` / `transferProgress` each fired at
   least once.
2. **Retry on wrong password.** First invocation of `credentials` returns a
   wrong token; second returns the real one. libgit2 is documented to
   retry; assert that the closure is called twice and `fetch` ultimately
   succeeds.
3. **Reject on bad certificate.** Point at a host with a known bad
   certificate (or simulate by returning `.reject` from
   `certificateCheck`). Assert `fetch` throws with `.callback` /
   `.certificate` class.

### 7.4 ThreadSanitizer

`RemoteConcurrencyTests` mirrors the v0.5a-ii `RebaseConcurrencyTests`
shape: two `Task`s starting `fetch` calls against the same `Repository`
should not produce TSan reports. Run as
`swift test --sanitize=thread --filter RemoteConcurrency`.

## 8. Errors

All public APIs are `throws(GitError)` (typed throws). Empirical libgit2
codes observed during implementation:

| API | Likely (`Code`, `Class`) | Scenario |
|---|---|---|
| `createRemote(named:url:)` | `.invalidSpec`, `.config` | illegal remote name |
| `createRemote(named:url:)` | `.exists`, `.config` | duplicate remote |
| `lookupRemote(named:)` | `.notFound`, `.config` | unknown remote |
| `deleteRemote(named:)` | `.notFound`, `.config` | unknown remote |
| `renameRemote(from:to:)` | `.invalidSpec` / `.exists`, `.config` | illegal new name / collision |
| `fetch(...)` | `.generic` / varies, `.net` | DNS / TCP / TLS / HTTP |
| `fetch(...)` | `.auth`, `.http` | 401/403 after credential retries exhausted |
| `fetch(...)` | `.user`, `.callback` | Swift closure threw or transferProgress returned false |
| `defaultBranch()` | `.unbornBranch` / `.notFound`, `.reference` | no HEAD on remote / HEAD not yet fetched |

Exact code/class pairs are confirmed empirically during implementation.

### 8.1 Throwing-closure error propagation

`throws(GitError)` on the public API forbids re-throwing arbitrary `Error`
types from Swift closures. The bridge normalizes:

- Closure throws `GitError` → re-thrown verbatim.
- Closure throws any other `Error` → wrapped as
  `GitError(code: .user, class: .callback, message: String(describing: error))`.

Callers that want to surface their own error type from the `credentials`
closure should conform to `GitError`-carrying contract, or accept the
wrapping with `.user` / `.callback` and inspect `message`.
