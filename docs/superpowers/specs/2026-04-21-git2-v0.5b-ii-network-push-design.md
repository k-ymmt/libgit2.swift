# Git2 Swift Wrapper — v0.5b-ii Network (Push) Slice

**Status:** Draft
**Date:** 2026-04-21

porcelain, the supporting `Repository.PushOptions` value type, and the two

(`docs/superpowers/specs/2026-04-20-git2-swift-wrapper-foundation-design.md`),
v0.3 (`docs/superpowers/specs/2026-04-20-git2-v0.3-read-extensions-design.md`),

## 1. Purpose

v0.5b-i delivered the read half of network I/O — `git_remote_fetch` plus the
`git_remote_callbacks` bridge for three Swift closures
(`credentials`, `certificate_check`, `transfer_progress`). What is still
missing is the write half: uploading local objects and advancing refs on a
remote Git host.

v0.5b-ii closes that gap:

- **Authenticated HTTPS push.** `Remote.push(...)` wrapping `git_remote_push`
  with Swift closures for `credentials`, `certificateCheck`,
  `pushTransferProgress`, and `pushUpdateReference`.
- **Per-ref result reporting with failure propagation.** Server-side rejections
  (non-fast-forward, pre-receive hook rejection, protected-branch rules) are
  collected via `push_update_reference` during the call and re-surfaced as a
  `GitError` after `git_remote_push` returns, so callers that do not supply a
  `pushUpdateReference` closure still see rejections as exceptions.
- **Refspec-driven push.** `push(refspecs:options:)` for callers who need to
  override the remote's configured push refspecs (e.g. force push via a
  refspec with the `+` prefix, delete-remote-ref via `:refs/heads/foo`).

v0.5b-ii is **additive**: every v0.2–v0.5b-i public API keeps the same
signature and behavior. The `RemoteCallbackBridge` internal module is
extended, not replaced.

## 2. Target Platforms

Unchanged from v0.5b-i: Apple platforms only, macOS 13+, iOS 16+, Swift 6.2.
HTTPS transport uses SecureTransport (selected when the v0.1 XCFramework was
built with `USE_HTTPS=SecureTransport`); no SSH (`USE_SSH=OFF`).

### 3.2 Explicitly not included in v0.5b-ii

Tracked in TODO.md under "Deferred from v0.5b-ii".

- **`push_negotiation` callback.** Surfacing this requires wrapping
  `git_push_update *updates` (array of `(src_refname, dst_refname, src_oid,
  dst_oid)`) as a Swift value type. Deferred because the two callbacks in
  scope — `push_transfer_progress` and `push_update_reference` — cover
  progress UI and per-ref result reporting respectively; `push_negotiation`
  is a niche "veto the whole push before transfer" hook that can be added
  additively once a concrete use case appears (e.g. pre-flight force-push
  confirmation UI).
- **`pb_parallelism`.** `git_push_options.pb_parallelism` controls the
  pack-builder thread count. Default 0 means libgit2 picks automatically;
  tuning requirement has not appeared in any downstream user of v0.5a / v0.5b-i.
- **`remote_push_options`.** Protocol-v2 server-side options passed as
  `-o key=value`. Used for things like `ci.skip=true` against GitLab runners.
  Downstream user base skews toward GitHub which does not honor these
  server-side options today. Additive when demand appears.
- **`git_remote_upload` / `_connect` / `_push` primitives.** v0.5b-ii ships
  only the porcelain `push`, matching v0.5b-i's "porcelain only" stance.
  Decomposed primitives land if a caller demonstrates a concrete need.
- **Proxy / SSH / SHA-256 / anonymous remotes.** All carried forward from
  v0.5b-i's deferred list. No change.

### 3.3 Non-goals (unchanged from v0.5b-i)

- No shipped CLI.
- No 1:1 exposure of every libgit2 function; low-level access remains via
  `@_exported import Cgit2`.
- No Linux / Windows / Android support.
- No Keychain / credential-UI integration — credentials are Swift values the
  caller supplies.
- No SHA-256 repositories.

## 4. Architectural Continuity

Every v0.2–v0.5b-i decision carries forward unchanged.

### 4.1 Handle ownership

`Remote` is the v0.5b-i handle: `final class`, `@unchecked Sendable`, retains
its parent `Repository` strongly, frees the underlying `git_remote *` in
`deinit`. `push` is a method addition only; no new handle types.

### 4.2 Per-`Repository` serialization

`Remote.push(...)` takes `repository.lock.withLock` once at the top and holds
it across the entire network round-trip: ref discovery, pack upload, server
response, `refs/remotes/<name>/*` fast-forward. A parallel `fetch` or `push`
on the same `Repository` blocks until the lock releases. This matches the
"one-operation-at-a-time-per-Repository" contract established in v0.2 and
preserved through v0.5b-i.

### 4.3 Callback re-entrancy

The four user-facing callbacks (`credentials`, `certificateCheck`,
`pushTransferProgress`, `pushUpdateReference`) run **synchronously on the
push thread while the repository lock is held**. A closure that reaches back
into `repository.*` (e.g. to read a config value) deadlocks on the
`HandleLock`'s `os_unfair_lock`. Every typealias's doc comment explicitly
warns against this. A live negative-path test is deliberately **not**
included — a deterministic deadlock would freeze the test runner and
require out-of-band timeout infrastructure that is not worth the cost for
a condition already surfaced by doc comments and by the `os_unfair_lock`
symptoms in the field.

UI updates from `pushTransferProgress` must be dispatched to the main actor
asynchronously (`Task { @MainActor in ... }`) — not done synchronously in
the closure body.

### 4.4 Per-ref partial failure → single `throws(GitError)`

libgit2's `git_remote_push` returns `0` even when the server rejects
individual refs. Per-ref outcomes arrive through the
`push_update_reference` callback as `(refname, status)` where `status == nil`
means accepted and `status != nil` is the server-supplied rejection reason
(e.g. `"non-fast-forward"`, `"pre-receive hook declined"`).

To preserve the v0.2 "failures are exceptions" contract, v0.5b-ii:

1. Collects every non-`nil` status into `ctx.pushRejections: [(String, String)]`
   during the call.
2. Forwards the tuple to the user's `pushUpdateReference` closure (if any)
   on every invocation — accepted and rejected alike.
3. After `git_remote_push` returns:
   - If `ctx.capturedError != nil` → throw that (fetch-equivalent error
     propagation, see §4.5).
   - Else if libgit2's return code is non-zero → `check(rc)` throws.
   - Else if `ctx.pushRejections` is non-empty → synthesize
     `GitError(code: .user, class: .reference, message: "rejected: <refname>: <status>; ...")`
     and throw. `.callback` is **not** a valid `GitError.Code` case in the current
     Git2 surface (it exists only on `Class`); `.user` semantically carries the
     same "our callback observed this" intent and parallels the fetch-side
     `GIT_EUSER` mapping.
   - Else return normally.

**Transport-dependent behavior.** The rejection-synthesis block described
above is only reachable when `push_update_reference` is the sole signal
libgit2 has that a ref was rejected — i.e. HTTP / SSH transports where
the remote sends per-ref status lines and libgit2 forwards them through
the callback. Over **local (`file://`) transport**, non-fast-forward is
rejected *directly* from `git_remote_push` with
`GIT_ENONFASTFORWARD` (surfaced as `GitError(code: .nonFastForward,
class: .reference, ...)`) — `push_update_reference` is not invoked and
`pushRejections` stays empty. Both paths are intentional and tested:
`RemotePushTests` covers the local-transport direct error
(`push_nonFastForward_localTransport_throwsNonFastForward`),
`RemoteGitHubIntegrationTests` covers the HTTP synthesis path.

Callers that want per-ref telemetry (including successes) supply a
`pushUpdateReference` closure. Callers that only care about pass/fail rely
on the synthesized throw.

The `push_update_reference` trampoline MUST return `0` for every
invocation — including rejected refs — so that libgit2 delivers results for
every ref in the refspec list before `git_remote_push` unwinds. Returning
non-zero short-circuits the call and hides rejection details for remaining
refs. This subtlety is captured in a source-level comment on the trampoline
(see §6.3).

### 4.5 Swift-closure → C-callback bridge (extension of v0.5b-i §4.5)

v0.5b-i's `RemoteCallbackBridge` is extended in place, not replaced. The
`RemoteCallbackContext` internal class gains three push-specific fields and
an error-capture slot is re-used (fetch and push never run concurrently on
the same context — each call site allocates its own):

```swift
internal final class RemoteCallbackContext {
    // fetch (v0.5b-i, unchanged)
    let credentials:       Repository.FetchOptions.CredentialsHandler?
    let certificateCheck:  Repository.FetchOptions.CertificateCheckHandler?
    let transferProgress:  Repository.FetchOptions.TransferProgressHandler?
    // push (v0.5b-ii, new)
    let pushTransferProgress: Repository.PushOptions.PushTransferProgressHandler?
    let pushUpdateReference:  Repository.PushOptions.PushUpdateReferenceHandler?
    var pushRejections: [(String, String)] = []
    // shared
    var capturedError: (any Error)?

    init(fetch options: Repository.FetchOptions) { ... }
    init(push  options: Repository.PushOptions)  { ... }
}
```

`Credentials` and `CertificateCheck` closure typealiases on `PushOptions`
are declared as `typealias X = Repository.FetchOptions.X` so the same
context field and the same v0.5b-i trampolines serve both call paths —
zero duplication of the credentials / certificate-check bridge.

`withRemoteCallbacks` gains a second overload for `PushOptions` (identical
body shape to the fetch overload — populate the `git_remote_callbacks`
struct, hand to `body` inside `withExtendedLifetime(ctx) { ... }`).

**Trampoline lifetime.** Unchanged from v0.5b-i: the context is a
stack-local in the push method, `passUnretained`'d into the
`git_remote_callbacks` payload, and extended across the `git_remote_push`
call with `withExtendedLifetime`.

**Error capture and propagation.** Identical to v0.5b-i:

- Closure throws `GitError` → re-thrown verbatim.
- Closure throws any other `Error` → wrapped as
  `GitError(code: .user, class: .callback, message: String(describing: error))`.
- `pushTransferProgress` returns `false` → trampoline returns
  `Int32(GIT_EUSER.rawValue)`, libgit2 aborts, Swift side surfaces the
  error as `GitError(code: .user, ...)` via `check(rc)` → `fromLibgit2`
  (libgit2 populates `git_error_last` with the `GIT_ERROR_CALLBACK`
  class on its way out).

### 4.6 `@Sendable` closures in a `Sendable` struct

Same rules as `FetchOptions` in v0.5b-i §4.6: `PushOptions` is `Sendable`,
all four closure typealiases are `@Sendable`, so `PushOptions` values cross
actor boundaries freely but closures cannot capture non-`Sendable` state.

### 4.7 Bare repository / unborn HEAD

`git_remote_push` does not touch HEAD — the caller names the refs to push
via `refspecs`. Bare / unborn-HEAD repositories push normally. libgit2
behavior surfaced unchanged.

## 5. Public API

### 5.1 `Remote.push`

```swift
extension Remote {
    /// Wraps `git_remote_push`.
    ///
    /// - Parameters:
    ///   - refspecs: Pass `nil` to use the remote's configured push refspecs
    ///     (most remotes have none configured and will push nothing on `nil`).
    ///     Pass `[Refspec("refs/heads/main:refs/heads/main")]` for a normal
    ///     push, `[Refspec("+refs/heads/main:refs/heads/main")]` for force
    ///     push, or `[Refspec(":refs/heads/obsolete")]` to delete a remote
    ///     ref.
    ///   - options: Push options. See `Repository.PushOptions`.
    /// - Throws: `GitError`. Per-ref rejections from the server (non-fast-forward,
    ///   pre-receive hook rejection, protected-branch rules) are collected
    ///   during the call and surfaced as
    ///   `GitError(code: .user, class: .reference)` with a semicolon-
    ///   delimited `refname: status` message. Callers that need programmatic
    ///   per-ref results should set `options.pushUpdateReference` and collect
    ///   state inside that closure.
    public func push(
        refspecs: [Refspec]? = nil,
        options: Repository.PushOptions = Repository.PushOptions()
    ) throws(GitError)
}
```

### 5.3 `Repository.PushOptions`

```swift
extension Repository {
    public struct PushOptions: Sendable {
        public typealias CredentialsHandler      = FetchOptions.CredentialsHandler
        public typealias CertificateCheckHandler = FetchOptions.CertificateCheckHandler

        /// Called repeatedly during pack upload.
        /// - Parameters:
        ///   - current: Objects packed and sent so far.
        ///   - total: Total objects the pack will contain.
        ///   - bytes: Cumulative bytes sent over the wire.
        /// - Returns: `true` to continue, `false` to cancel. On cancel, the
        ///   enclosing `push` throws
        ///   `GitError(code: .user, class: .callback)`.
        ///
        /// - Important: Runs synchronously on the push thread with the
        ///   repository lock held. Do **not** call `repository.*` from
        ///   inside — doing so deadlocks on `os_unfair_lock`. Dispatch UI
        ///   updates via `Task { @MainActor in ... }` rather than synchronous
        ///   main-actor calls.
        public typealias PushTransferProgressHandler =
            @Sendable (_ current: Int, _ total: Int, _ bytes: Int) -> Bool

        /// Called once per ref after the server responds.
        /// - Parameters:
        ///   - refname: The ref the push targeted on the remote.
        ///   - status: `nil` means the server accepted the update. Non-`nil`
        ///     is the server-supplied rejection reason
        ///     (e.g. `"non-fast-forward"`).
        ///
        /// - Important: Same re-entrancy rules as
        ///   `PushTransferProgressHandler`.
        public typealias PushUpdateReferenceHandler =
            @Sendable (_ refname: String, _ status: String?) -> Void

        public var credentials:           CredentialsHandler?
        public var certificateCheck:      CertificateCheckHandler?
        public var pushTransferProgress:  PushTransferProgressHandler?
        public var pushUpdateReference:   PushUpdateReferenceHandler?

        public var followRedirects: FetchOptions.RedirectPolicy = .initial
        public var customHeaders:   [String]                    = []

        public init()
    }
}
```

**Field choices (why these and not others):**

- `pushTransferProgress` takes `(current: Int, total: Int, bytes: Int) -> Bool`
  directly, not a `TransferProgress` value type. libgit2's
  `push_transfer_progress_cb` signature is
  `(uint32_t current, uint32_t total, size_t bytes, void *payload)` — it
  does not populate the `git_indexer_progress` structure that fetch uses
  (no delta count, no indexed/received split, no localObjects). Re-using
  `TransferProgress` would require zero-padding those fields, which is
  misleading. A dedicated 3-arg signature stays honest.
- `pushUpdateReference` surfaces `String?` for status rather than a
  `PushStatus` enum. libgit2 passes through the server's response string
  verbatim — there is no canonical set of status codes in the Git push
  protocol. Callers that want to branch on `"non-fast-forward"` vs other
  strings can do so; v0.5b-ii does not attempt to enumerate them.
- `pb_parallelism` and `remote_push_options` are not exposed. See §3.2.
- `followRedirects` and `customHeaders` ride the same types as fetch
  (`FetchOptions.RedirectPolicy`, `[String]`) since the underlying
  `git_push_options` and `git_fetch_options` fields are identical.

### 6.3 New trampolines

```swift
private func remoteBridge_pushTransfer(
    current: UInt32, total: UInt32, bytes: Int,
    payload: UnsafeMutableRawPointer?
) -> Int32 {
    guard let payload else { return -1 }
    let ctx = Unmanaged<RemoteCallbackContext>
        .fromOpaque(payload).takeUnretainedValue()
    guard let handler = ctx.pushTransferProgress else { return 0 }
    return handler(Int(current), Int(total), bytes) ? 0 : -1
}

private func remoteBridge_pushUpdateReference(
    refname: UnsafePointer<CChar>?,
    status:  UnsafePointer<CChar>?,
    payload: UnsafeMutableRawPointer?
) -> Int32 {
    guard let payload, let refname else { return -1 }
    let ctx = Unmanaged<RemoteCallbackContext>
        .fromOpaque(payload).takeUnretainedValue()
    let name = String(cString: refname)
    let statusStr = status.map(String.init(cString:))
    if let s = statusStr { ctx.pushRejections.append((name, s)) }
    ctx.pushUpdateReference?(name, statusStr)
    // Return 0 on every invocation — including rejected refs — so libgit2
    // continues delivering results for the remaining refspecs before
    // git_remote_push unwinds. Returning non-zero here short-circuits the
    // call and hides rejection details for refs we have not seen yet.
    // The Swift side inspects ctx.pushRejections after git_remote_push
    // returns and synthesizes a single GitError covering all rejected refs.
    return 0
}
```

The `return 0` comment is **mandatory** in the final source — it documents
the non-obvious coupling between the trampoline's return value and §4.4's
rejection-collection contract.

### 6.4 `Remote.push` implementation sketch

Mirrors the shape of the existing `Remote.fetch` body (see
`Sources/Git2/Objects/Remote.swift`): one `repository.lock.withLock`, one
`withRemoteCallbacks`, nested `withGitStrArray` for refspecs and custom
headers, post-call error triage.

```swift
extension Remote {
    public func push(
        refspecs: [Refspec]? = nil,
        options: Repository.PushOptions = Repository.PushOptions()
    ) throws(GitError) {
        try repository.lock.withLock { () throws(GitError) in
            try withRemoteCallbacks(options) { cbsPtr, ctx throws(GitError) in
                var pushOpts = git_push_options()
                let initRC = git_push_options_init(&pushOpts, UInt32(GIT_PUSH_OPTIONS_VERSION))
                precondition(initRC == 0)
                pushOpts.callbacks        = cbsPtr.pointee
                pushOpts.follow_redirects = options.followRedirects.asGit

                let specs = refspecs?.map(\.string) ?? []
                let rc: Int32 = try withGitStrArray(specs) { (specsPtr: UnsafePointer<git_strarray>?) throws(GitError) -> Int32 in
                    return try withGitStrArray(options.customHeaders) { (headersPtr: UnsafePointer<git_strarray>?) throws(GitError) -> Int32 in
                        if let headersPtr {
                            pushOpts.custom_headers = headersPtr.pointee
                        }
                        return git_remote_push(self.handle, specsPtr, &pushOpts)
                    }
                }

                // Order matters: captured closure error > libgit2 rc > collected rejections.
                if let captured = ctx.capturedError {
                    if let gitErr = captured as? GitError { throw gitErr }
                    throw GitError(
                        code: .user,
                        class: .callback,
                        message: String(describing: captured)
                    )
                }
                try check(rc)
                if !ctx.pushRejections.isEmpty {
                    let detail = ctx.pushRejections
                        .map { "\($0.0): \($0.1)" }
                        .joined(separator: "; ")
                    throw GitError(
                        code: .user,
                        class: .reference,
                        message: "rejected: \(detail)"
                    )
                }
            }
        }
    }
}
```

The `withRemoteCallbacks` overload for `PushOptions` mirrors v0.5b-i's
fetch overload: `(UnsafeMutablePointer<git_remote_callbacks>,
RemoteCallbackContext) throws(GitError) -> R` — the `throws(GitError)` on
the body closure lets the refspec / custom-headers nested helpers forward
`GitError` upward without the `Result<R, GitError>` dance in
`withGitStrArray` (the inner helpers already handle that themselves).

### 7.3 GitHub integration — push extension

`ENABLE_GITHUB_INTEGRATION_TESTS=1`. Re-uses the existing
`setup-github-fixture.sh` output (`repo_url`, `token`, `cleanup_name`). Test
body:

1. `git_clone`-equivalent via v0.5b-i `fetch` (or raw clone helper) of
   `repo_url` using the token as password.
2. Create a local commit (empty tree + new commit object, author from
   `Signature.now(name:email:)`).
3. `push(remoteNamed: "origin", refspecs: [Refspec("refs/heads/main:refs/heads/main")], options: optionsWithCreds)`.
4. Verify via `gh api repos/:owner/$REPO/commits/main` that the HEAD OID
   matches the pushed commit.
5. For the non-fast-forward case: after step 3 succeeds, reset the local
   branch to `HEAD~1`, commit something else, and push **without** the `+`
   prefix. Assert `GitError(.user, .reference)` with message containing
   `"non-fast-forward"`.
6. For the auth-failure case: push with a `credentials` closure returning a
   deliberately invalid token. Assert `.auth` / `.http`.
7. `teardown-github-fixture.sh $cleanup_name` regardless of test outcome.

Orphan-repo risk on teardown failure is a pre-existing v0.5b-i concern;
no change.

### 7.4 ThreadSanitizer

`RemotePushConcurrencyTests` runs as a normal smoke test under standard
`swift test`. TSan-under-`swift test` remains blocked on the macOS 25.3 /

## 8. Errors

All public APIs are `throws(GitError)`. Empirical libgit2 codes observed
during implementation:

| API | Likely `(Code, Class)` | Scenario |
|---|---|---|
| `push(remoteNamed:)` sugar | `.notFound`, `.config` | unknown remote |
| `push(...)` | `.invalidSpec`, `.net` or `.reference` | malformed refspec |
| `push(...)` | `.generic` / varies, `.net` | DNS / TCP / TLS |
| `push(...)` | `.auth`, `.http` | 401/403 after credential retries exhausted |
| `push(...)` | `.user`, `.callback` | Swift closure threw or `pushTransferProgress` returned `false` |
| `push(...)` | `.user`, `.certificate` | `certificateCheck` returned `.reject` |
| `push(...)` over HTTP/SSH | `.user`, `.reference` | server rejected one or more refs (pre-receive hook, protected branch) — via `push_update_reference` callback + synthesis |
| `push(...)` over `file://` | `.nonFastForward`, `.reference` | non-fast-forward push rejected directly by libgit2's local transport; `push_update_reference` is not invoked |

Exact code/class pairs are confirmed empirically during implementation; spec
values are best-effort and updated if observed behavior diverges.

### 8.1 Rejection message format

`"rejected: <refname1>: <status1>; <refname2>: <status2>"`. Semicolon-
delimited, stable enough for CI log pattern-matching but explicitly **not**
a machine-readable contract. Callers that need structured per-ref results
MUST use `pushUpdateReference`. The doc comment on `Remote.push(...)`
states this in one sentence.

### 8.2 Throwing-closure propagation

Identical to v0.5b-i §8.1:

- Closure throws `GitError` → verbatim.
- Closure throws any other `Error` → wrapped as
  `GitError(code: .user, class: .callback, message: String(describing: error))`.
