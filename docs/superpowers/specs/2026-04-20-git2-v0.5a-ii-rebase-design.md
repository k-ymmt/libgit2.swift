**Status:** Draft

## 1. Purpose

v0.5a-i closed the non-stateful integration cases (merge primitives, stateful
merge + porcelain, cherry-pick). What is still missing is **rebase** — the
commit-replay pipeline that is the natural companion to merge and cherry-pick.

Rebase is unique among the integration operations in that it is **stateful
across process boundaries**: libgit2 persists the in-progress rebase under
`.git/rebase-merge/`, and `git_rebase_open` can pick up a rebase started by
either an earlier process or another libgit2 client. This cross-process
resumability is what v0.5a-i named as the reason rebase warrants its own slice.

v0.5a-ii delivers three use cases jointly:

- **Rebase-as-a-library, single-process.** A caller invokes
  `startRebase(branch:upstream:onto:)`, loops `while let op = try rebase.next()
  { try rebase.commit(committer:) }`, and finishes with `rebase.finish()`.
- **Resume a rebase across processes.** `openRebase()` picks up a rebase that
  a previous process left half-applied (e.g. after a conflict was resolved by
  hand).
- **In-memory rebase for preview / automation.** `RebaseOptions.inMemory =
  true` replays commits without touching HEAD or the working tree; the result
  index is surfaced via `rebase.inMemoryIndex()`. Useful for "would this
  rebase cleanly?" preview UIs and for non-interactive automation that wants
  to build a rebased tree off the working directory.

v0.5a-ii is **additive**: every existing v0.2 / v0.3 / v0.4a / v0.4b-i /
v0.4b-ii / v0.5a-i public API keeps the same signature and behavior.

### 3.2 Explicitly not included in v0.5a-ii

Tracked in TODO.md under "Deferred from v0.5a-ii".

- **Rebase callbacks (`commit_create_cb` / `signing_cb`).** Same deferral
  pattern as v0.4b-ii's checkout callbacks and v0.5a-i's merge callbacks — the
  Swift-closure-to-C-function-pointer bridge (context pointer lifetime,
  cancellation propagation, typed-throws leakage) is its own implementation
  surface. `signing_cb` is additionally deprecated in libgit2. Callers who
  need signed rebased commits compose `@_exported import Cgit2` until a
  unified callbacks slice lands.
- **`Commit` / `Reference`-accepting `startRebase` overloads.** v0.5a-ii
  requires explicit `AnnotatedCommit`; convenience overloads can land
  additively later. Matches v0.5a-i's policy on merge / cherry-pick.
- **Interactive rebase todo-list manipulation.** libgit2 does not publish a
  `rebase-todo` editing API. v0.5a-ii surfaces the operation list for
  inspection (`operation(at:)`) but not mutation. If libgit2 grows a public
  editing API, a follow-up slice can surface it.
- **Porcelain `rebase(onto:)` with auto-continue.** v0.5a-ii stops at the
  primitives. A higher-level "apply the whole rebase, throw if conflicts"
  wrapper can land later additively.
- **`abortRebase()` convenience on `Repository`.** Composable today via
  `openRebase()` + `rebase.abort()` if a concrete pattern emerges.

## 4. Architectural Continuity

Every v0.2 / v0.3 / v0.4a / v0.4b-i / v0.4b-ii / v0.5a-i decision carries
forward unchanged.

### 4.1 Handle ownership

`Rebase` joins the existing pattern: a `final class`, `@unchecked Sendable`,
retaining its parent `Repository` strongly and freeing the underlying
`git_rebase *` in `deinit`. Creation happens only through two `Repository`
extension methods (`startRebase(...)`, `openRebase(...)`). Serialization is
the parent repository's `HandleLock`.

### 4.2 Per-`Repository` serialization

Every new method takes `repository.lock.withLock` once at the top. This
includes the `Rebase` instance methods (`next`, `commit`, `finish`, `abort`,
`operationCount`, `currentOperationIndex`, `operation(at:)`, `origHeadName`,
`origHeadOid`, `ontoName`, `ontoOid`, `inMemoryIndex`).

The lock is **not** held across `next` / `commit` pairs. Callers holding a
`Rebase` instance for an extended period (e.g. a UI that waits for the user
to resolve a conflict) do **not** block other repository operations on the
same `Repository`. libgit2 serializes its own reads of
`.git/rebase-merge/state` under the hood, and the cross-process resumability
story (via `git_rebase_open`) means the Swift layer does not need to hold
exclusive access across the whole rebase session.

### 4.3 Annotated-commit provenance

`startRebase(branch:upstream:onto:)` takes three `AnnotatedCommit`s, all
optional (libgit2's nullable `branch` / `upstream` / `onto` semantics are
surfaced verbatim). The provenance carried by each `AnnotatedCommit` —
whether it was resolved from a local branch, from `FETCH_HEAD`, or from a raw
OID — is what drives reflog messages (e.g. `rebase: checkout refs/heads/main`
rather than just `rebase: checkout <oid>`). This is the same rationale v0.5a-i
§4.5 gave for exposing `AnnotatedCommit` rather than synthesizing provenance
under the covers.

### 4.4 State model

Rebase state (`.rebase` / `.rebaseInteractive` / `.rebaseMerge`) is already
covered by v0.5a-i's `Repository.State` enum and `cleanupState()`. v0.5a-ii
does **not** touch those APIs — once a `Rebase` session is in progress,
`repository.state` returns the appropriate case; `cleanupState()` removes the
state files (but does **not** reset the working tree, same as for merge /
cherry-pick). The intended abort flow is `rebase.abort()`, which libgit2
implements with an implicit `GIT_CHECKOUT_FORCE`.

### 4.5 Bare repository

libgit2's `git_rebase_init` with `inmemory == false` rejects bare
repositories through its internal working-tree checkout path.
`inmemory == true` does not need a working tree and succeeds on bare repos.
v0.5a-ii does not add a Swift-layer pre-check; libgit2's errors propagate
unchanged. Matches v0.5a-i's policy.

### 4.6 Unborn HEAD

`git_rebase_init` on an unborn HEAD returns a libgit2 error; v0.5a-ii
propagates it unchanged. The empirical `code` / `class` values are confirmed
during implementation (§9.4).

### 4.7 Handle re-use after termination

`rebase.finish()` / `rebase.abort()` consume the rebase session on the
repository side: libgit2 removes `.git/rebase-merge/`. The Swift `Rebase`
handle remains valid until `deinit`, and a subsequent call to `next` /
`commit` / `finish` / `abort` returns whatever libgit2 returns in that state
(typically a failure). v0.5a-ii does **not** add Swift-layer state tracking
to reject the second call — the thin-wrapper convention (matching
`cleanupState()`, `removeMessage()`, etc.) is maintained. See §10.1 for the
empirical error record.

### 4.8 Iterator terminator

`git_rebase_next` returns `GIT_ITEROVER` once every operation has been
applied. v0.5a-ii translates that to Swift `Optional.none`, matching the v0.2
`CommitIterator.next()` convention. All other libgit2 errors throw normally.
This lets callers write the natural `while let op = try rebase.next() { … }`
loop.

### 4.9 In-memory vs on-disk

`RebaseOptions.inMemory = true` maps to `git_rebase_options.inmemory = 1`.
When set, libgit2 does **not** rewind HEAD, does **not** write
`.git/rebase-merge/`, does **not** touch the working tree. The per-operation
result index is retrieved via `rebase.inMemoryIndex()`, which wraps
`git_rebase_inmemory_index`. Calling `inMemoryIndex()` on an on-disk rebase
surfaces libgit2's error unchanged.

### 4.10 Atomicity

Individual libgit2 calls (`git_rebase_next`, `git_rebase_commit`,
`git_rebase_finish`, `git_rebase_abort`) are each executed inside one lock
section. The Swift layer does not batch multiple libgit2 calls into a single
critical section on the rebase path — neither the natural API shape nor the
cross-process resumability requires it. If a `commit` call fails halfway
through (e.g. disk full), the on-disk `.git/rebase-merge/` state remains
whatever libgit2 left it, and the caller can resume (or abort) via
`openRebase()` in a later session. Same trade-off shape as v0.5a-i §4.8.

## 6. Types

### 6.1 `Rebase`

```swift
// Sources/Git2/Objects/Rebase.swift

import Cgit2

/// A libgit2 rebase session.
///
/// A `Rebase` handle wraps `git_rebase *`. It represents an in-progress
/// rebase: the operation list, the current index, and the reflog prefix.
/// Instances are produced by ``Repository/startRebase(branch:upstream:onto:options:)``
/// (start a fresh rebase) or ``Repository/openRebase(options:)`` (resume a
/// rebase started by this or another process via `.git/rebase-merge/`).
///
/// A typical single-process flow:
///
/// ```swift
/// let rebase = try repo.startRebase(upstream: try repo.annotatedCommit(for: upstreamRef))
/// while let op = try rebase.next() {
///     // … resolve conflicts if any …
///     _ = try rebase.commit(committer: signature)
/// }
/// try rebase.finish()
/// ```
///
/// After ``finish(signature:)`` or ``abort()`` returns successfully, the
/// repository no longer has a rebase in progress. The Swift handle remains
/// valid until `deinit` frees it, but subsequent calls to iteration /
/// termination methods surface libgit2 errors unchanged — v0.5a-ii does not
/// add a Swift-side "consumed" guard.
public final class Rebase: @unchecked Sendable {
    internal let handle: OpaquePointer

    /// The repository this rebase belongs to.
    public let repository: Repository

    internal init(handle: OpaquePointer, repository: Repository) {
        self.handle = handle
        self.repository = repository
    }

    deinit {
        git_rebase_free(handle)
    }
}
```

### 6.2 `RebaseOperation`

```swift
// Sources/Git2/Values/RebaseOperation.swift

/// One step in a rebase operation list. Value-type snapshot of
/// `git_rebase_operation`.
public struct RebaseOperation: Sendable, Equatable {
    /// The operation kind.
    public let kind: Kind

    /// The commit being cherry-picked. Compares equal to ``OID/zero`` for
    /// `.exec` operations — libgit2's `git_rebase_operation.id` is undefined
    /// in that case. Callers should branch on ``kind`` rather than inspect
    /// `oid` for `.exec`.
    public let oid: OID

    /// The executable command string for `.exec` operations; `nil` for every
    /// other kind.
    public let exec: String?

    /// Operation kind. Mirrors `git_rebase_operation_t`.
    public enum Kind: Sendable, Equatable {
        case pick        // GIT_REBASE_OPERATION_PICK
        case reword      // GIT_REBASE_OPERATION_REWORD
        case edit        // GIT_REBASE_OPERATION_EDIT
        case squash      // GIT_REBASE_OPERATION_SQUASH
        case fixup       // GIT_REBASE_OPERATION_FIXUP
        case exec        // GIT_REBASE_OPERATION_EXEC
    }
}
```

### 6.3 `Repository.RebaseOptions`

```swift
// Sources/Git2/Values/RebaseOptions.swift

extension Repository {
    /// Options forwarded to libgit2's `git_rebase_init` and `git_rebase_open`.
    ///
    /// The default value (`RebaseOptions()`) maps to libgit2's defaults —
    /// `quiet = false`, `inMemory = false`, no rewrite-notes ref, default
    /// merge / checkout options.
    public struct RebaseOptions: Sendable, Equatable {
        /// Advisory "quiet" hint. libgit2 itself does not act on this flag;
        /// it is passed through for interoperability with other Git tools.
        public var quiet: Bool

        /// When `true`, runs the rebase entirely in memory — HEAD is not
        /// rewound, `.git/rebase-merge/` is not written, and the working
        /// tree is not touched. Pair with ``Rebase/inMemoryIndex()`` to read
        /// the per-operation result.
        public var inMemory: Bool

        /// When non-nil, `git_rebase_finish` rewrites notes (in the named
        /// ref) for the rebased commits. When `nil`, libgit2 consults the
        /// `notes.rewriteRef` / `notes.rewrite.rebase` config keys and
        /// defaults to not rewriting.
        public var rewriteNotesRef: String?

        /// Passed to `git_rebase_next` to control the 3-way merge used during
        /// each patch application.
        public var merge: MergeOptions

        /// Passed to `git_rebase_init`, `git_rebase_next`, and
        /// `git_rebase_abort`. libgit2 implicitly adds `GIT_CHECKOUT_FORCE`
        /// on `abort` to match `git` semantics, regardless of the value set
        /// here.
        public var checkout: CheckoutOptions

        public init(
            quiet: Bool = false,
            inMemory: Bool = false,
            rewriteNotesRef: String? = nil,
            merge: MergeOptions = MergeOptions(),
            checkout: CheckoutOptions = CheckoutOptions()
        ) {
            self.quiet = quiet
            self.inMemory = inMemory
            self.rewriteNotesRef = rewriteNotesRef
            self.merge = merge
            self.checkout = checkout
        }
    }
}
```

Fields **not** surfaced: `commit_create_cb`, `signing_cb`, `payload`,
`reserved`. Tracked in §10.2.

## 7. Public API

### 7.1 Lifecycle

```swift
// Sources/Git2/Repository/Repository+Rebase.swift

extension Repository {
    /// Wraps `git_rebase_init`. Initializes a new rebase session.
    ///
    /// All three `AnnotatedCommit` arguments are optional, following
    /// libgit2's nullable semantics:
    /// - `branch: nil` → rebase the current branch
    /// - `upstream: nil` → rebase every commit reachable from `branch`
    /// - `onto: nil` → rebase onto `upstream`
    ///
    /// Call ``Rebase/next()`` to begin applying operations.
    public func startRebase(
        branch: AnnotatedCommit? = nil,
        upstream: AnnotatedCommit? = nil,
        onto: AnnotatedCommit? = nil,
        options: RebaseOptions = RebaseOptions()
    ) throws(GitError) -> Rebase

    /// Wraps `git_rebase_open`. Resumes an in-progress rebase that was
    /// started by this or another process (libgit2 persists rebase state
    /// under `.git/rebase-merge/`).
    ///
    /// - Throws: ``GitError/Code/notFound`` when no rebase is in progress.
    public func openRebase(
        options: RebaseOptions = RebaseOptions()
    ) throws(GitError) -> Rebase
}
```

### 7.2 Iteration

```swift
// Sources/Git2/Objects/Rebase.swift (extension)

extension Rebase {
    /// Wraps `git_rebase_next`. Applies the next operation — for any
    /// non-`.exec` kind the patch is applied to the index and (unless
    /// ``Repository/RebaseOptions/inMemory`` is `true`) to the working tree.
    /// Conflicts, if any, land in the index; callers inspect
    /// ``Index/hasConflicts`` + ``Index/conflicts()`` before calling
    /// ``commit(author:committer:message:encoding:)``.
    ///
    /// - Returns: the next ``RebaseOperation``, or `nil` when all operations
    ///   have been applied (libgit2 `GIT_ITEROVER` translated to
    ///   `Optional.none`).
    public func next() throws(GitError) -> RebaseOperation?

    /// Wraps `git_rebase_commit`. Commits the current patch.
    ///
    /// - Parameters:
    ///   - author: the author for the rebased commit. Passing `nil` keeps the
    ///     author of the original commit.
    ///   - committer: the committer for the rebased commit. Required —
    ///     identifies the rebaser.
    ///   - message: the commit message. Passing `nil` keeps the original
    ///     commit's message.
    ///   - encoding: the message encoding, per libgit2's convention (`nil`
    ///     keeps the original; `"UTF-8"` is the common explicit value).
    /// - Throws:
    ///   - ``GitError/Code/unmerged`` when the index still contains conflicts
    ///     from the most recent ``next()``.
    ///   - ``GitError/Code/applied`` when the current patch is already in
    ///     `upstream` and there is nothing to commit.
    @discardableResult
    public func commit(
        author: Signature? = nil,
        committer: Signature,
        message: String? = nil,
        encoding: String? = nil
    ) throws(GitError) -> OID
}
```

### 7.3 Termination

```swift
extension Rebase {
    /// Wraps `git_rebase_finish`. Finalizes a rebase after every operation
    /// has been applied. Removes `.git/rebase-merge/`. Optionally writes a
    /// reflog signature line.
    public func finish(signature: Signature? = nil) throws(GitError)

    /// Wraps `git_rebase_abort`. Aborts the rebase and restores the
    /// repository + working tree to their state before rebase began.
    /// libgit2 implicitly applies `GIT_CHECKOUT_FORCE` during this restore.
    public func abort() throws(GitError)
}
```

After either termination method returns successfully, the repository no
longer has a rebase in progress. The Swift `Rebase` handle remains valid
until `deinit`; subsequent calls to iteration / termination methods surface
libgit2's errors unchanged (see §4.7).

### 7.4 Metadata

## 8. Concurrency / Locking

Unchanged from v0.5a-i: every public entry point takes
`repository.lock.withLock` once at the top. `Rebase` instance methods take
their parent repository's lock.

The lock is **not** held across iterations: `rebase.next()` and
`rebase.commit(...)` each take-and-release the lock. This matches the
per-call locking of the low-level merge / cherry-pick primitives and avoids
holding the lock through a user-interactive conflict-resolution pause.

`Rebase` is `@unchecked Sendable` following the existing handle convention.
Its metadata accessors (`operationCount`, `currentOperationIndex`,
`operation(at:)`, `origHeadName`, `origHeadOid`, `ontoName`, `ontoOid`) read
libgit2 state that can change under `next` / `commit` — they take the lock.

Cross-process resumability is libgit2's responsibility: `.git/rebase-merge/`
is the source of truth, and `openRebase()` can read state a prior process
wrote. The Swift `HandleLock` does not mediate cross-process access (no
mechanism for that exists).

  — shared ancestor, `upstream` is `upstreamAhead` commits ahead, `feature`
  is branched off the ancestor and has `featureAhead` non-conflicting
  commits.
- `TestFixture.makeConflictingRebase(...) throws -> ...` — shared ancestor,
  both `upstream` and `feature` edit the same line of the same file.
- `TestFixture.makeAlreadyApplied(...) throws -> ...` — `feature`'s tip is
  already reachable from `upstream` (same tree), so `commit()` on the
  resulting rebase operation throws `.applied`.

### 9.4 Error-code specificity

Per v0.3 / v0.4a / v0.4b-i / v0.4b-ii / v0.5a-i policy, every `throws:` test
asserts `error.code` and `error.class` explicitly (no bare
`throws: GitError.self`). Empirical values for each failure path are
confirmed during implementation and recorded in the relevant test file
following the v0.4b-ii "Empirical" block pattern. Paths whose error codes
need empirical confirmation:

- `openRebase()` on a clean repo → expected `.notFound`.
- `commit()` on a conflicting index → expected `.unmerged`.
- `commit()` on an already-applied commit → expected `.applied`.
- `inMemoryIndex()` on a non-inmemory rebase.
- `next()` on a bare, non-inmemory rebase start.
- `startRebase()` on an unborn HEAD.
- Second call to `finish()` / `abort()` / `next()` / `commit()` on a
  terminated rebase.

### 9.5 Not covered

- **ThreadSanitizer / sustained fuzz.** Inherits the v0.4b-ii / v0.5a-i
  stance — basic "lock serializes" coverage via `RebaseConcurrencyTests`;
  TSan in CI remains on the post-v0.2.0 TODO list.
- **Cross-process rebase resume (real second process).** `openRebase()` is
  tested within a single Swift process (start → discard → open). Running
  `git rebase` from a shell in one step and `openRebase()` in another would
  be the higher-fidelity test; skipped because the CI shape is messy and the
  single-process version exercises the same libgit2 code path.
- **Rebase callbacks (`commit_create_cb` / `signing_cb`).** Not surfaced in
  v0.5a-ii.
- **Interactive rebase todo-list edits.** No public libgit2 API to exercise.
- **Atomic rollback mid-`commit()`.** If a `commit()` call is interrupted
  (SIGKILL, disk full), the on-disk `.git/rebase-merge/` state is whatever
  libgit2 left — no Swift-layer rollback. Callers recover via
  `openRebase()` + `abort()` in a later session.
- **FileMode integration coverage for `.link` / `.commit` through rebase.**
  Same shape as the v0.4a / v0.4b-ii / v0.5a-i deferred items — surfaces
  once the fixture layer supports those filemodes more conveniently.

## 10. Open Questions / Deferrals

### 10.1 Open questions (to resolve during implementation)

- **Empirical error codes.** Every entry in §9.4 needs its observed
  `GitError.Code` / `.Class` recorded in the relevant test file.
- **`operation(at:)` out-of-bounds behavior.** libgit2 documents a `NULL`
  return; v0.5a-ii maps that to Swift `nil`. Confirm libgit2 does not also
  set a thread-local error on the out-of-bounds path (which would make the
  `nil` return misleading).
- **`git_rebase_operation.id` for `.exec`.** libgit2's struct comment says
  `id` is populated "for all operations except those of type
  `GIT_REBASE_OPERATION_EXEC`". v0.5a-ii documents the field as `OID.zero` on
  `.exec` and asks callers to branch on ``kind``. Confirm during
  implementation that libgit2 actually zero-fills the field (the `const
  git_oid id` member is likely zero-initialized from the operation-list
  constructor, but worth checking). If it turns out to hold stale data, the
  doc comment is tightened to "value is undefined; do not inspect".
- **`withOptionalCString` helper.** No existing helper in
  `Sources/Git2/Internal/`. Add one as part of this slice: a `String?` →
  `UnsafePointer<CChar>?` closure-scoped bridge that passes `nil` through
  verbatim. Follow the pattern of existing bridges (see
  `StrArrayBridge.swift`).

### 10.2 Explicitly deferred (not landing in v0.5a-ii)

Tracked under "Deferred from v0.5a-ii" in TODO.md:

- **Rebase callbacks (`commit_create_cb` / `signing_cb`).** Same deferral as
  v0.4b-ii's checkout callbacks and v0.5a-i's merge callbacks.
- **`Commit` / `Reference`-accepting `startRebase` overloads.** Convenience
  overloads bypassing explicit `AnnotatedCommit`. Additive; can land later.
- **Porcelain `rebase(onto:)` with auto-continue.** Higher-level "apply the
  whole rebase, surface conflicts once at the end" wrapper.
- **`abortRebase()` on `Repository`.** Composable via `openRebase()` +
  `rebase.abort()` today.
- **Interactive rebase todo-list editing.** libgit2 does not publish the
  API surface.
- **Fidelity test for cross-process resume.** Non-trivial CI shape.

### 10.3 v0.5b linkage

v0.5b (remote / fetch / push) adds `git_annotated_commit_from_fetchhead`.
That annotated-commit construction route plugs into v0.5a-ii's
`startRebase(...)` as-is — no signature changes expected. The "rebase my
branch onto `FETCH_HEAD`" flow becomes a two-step script (`annotatedCommit`
from fetchhead → `startRebase(onto:)`) once v0.5b lands.
