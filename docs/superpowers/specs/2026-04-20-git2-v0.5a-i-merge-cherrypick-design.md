# Git2 Swift Wrapper — v0.5a-i Merge / Cherry-pick Slice

**Status:** Draft
**Date:** 2026-04-20
**Scope:** First slice of the v0.5a "integration operations" line. Adds the merge primitives (`git_annotated_commit`, `git_merge_base`, `git_merge_analysis`, `git_merge_trees`, `git_merge_commits`), the stateful merge entry points (`git_merge` — low-level and a `Reference`-taking porcelain), the cherry-pick family (`git_cherrypick`, `git_cherrypick_commit`), repository-state introspection (`git_repository_state`, `git_repository_message`, `git_repository_message_remove`, `git_repository_state_cleanup`), and the v0.4b-ii deferred `setHead(detachedAtAnnotated:)`. Builds on v0.2 (`docs/superpowers/specs/2026-04-20-git2-swift-wrapper-foundation-design.md`), v0.3 (`docs/superpowers/specs/2026-04-20-git2-v0.3-read-extensions-design.md`), v0.4a (`docs/superpowers/specs/2026-04-20-git2-v0.4a-write-foundation-design.md`), v0.4b-i (`docs/superpowers/specs/2026-04-20-git2-v0.4b-i-index-design.md`), and v0.4b-ii (`docs/superpowers/specs/2026-04-20-git2-v0.4b-ii-checkout-head-design.md`).

## 1. Purpose

v0.4a closed ODB write. v0.4b-i closed the index / staging surface. v0.4b-ii closed working-tree checkout and HEAD manipulation. What is still missing is the ability to **combine two (or more) lines of history into one** — either by replaying one side onto another (cherry-pick) or by performing a 3-way merge.

v0.5a-i closes that gap for the non-stateful cases. It does **not** cover rebase — rebase is stateful across process boundaries (`git_rebase_open` resumes a rebase started in an earlier process), which warrants its own slice (v0.5a-ii).

The slice delivers two use cases jointly:

- **Merge-as-a-library.** A caller can run the full `git merge <branch>` pipeline (analyze → fast-forward if appropriate, otherwise 3-way merge that writes MERGE_HEAD / MERGE_MSG) in a single high-level call, then either commit the result with the existing v0.4a `commit(...)` API or abort via `cleanupState()` + `checkoutHead(strategy: .force)`.
- **Custom tooling.** The pure-calculation primitives (`mergeBase`, `mergeTrees`, `mergeCommits`, `cherrypickCommit`) return an `Index` without touching the working tree, which is what UI tools that want to preview a merge, or non-interactive automation, need.

v0.5a-i is **additive**: every existing v0.2 / v0.3 / v0.4a / v0.4b-i / v0.4b-ii public API keeps the same signature and behavior.

## 2. Target Platforms

Unchanged from v0.4b-ii: Apple platforms only, macOS 13+, iOS 16+, Swift 6.2.

## 4. Architectural Continuity

Every v0.2 / v0.3 / v0.4a / v0.4b-i / v0.4b-ii decision carries forward unchanged.

### 4.1 Handle ownership

`AnnotatedCommit` joins the existing pattern: a `final class`, `@unchecked Sendable`, retaining its parent `Repository` strongly and freeing the underlying `git_annotated_commit *` in `deinit`. Creation happens only through `Repository` extension methods (see §7.1). Serialization is the parent repository's `HandleLock`.

### 4.2 Per-`Repository` serialization

Every new method takes `repository.lock.withLock` once at the top. Two multi-step operations matter:

- **Porcelain `merge(_ branch:)`.** The `peel ref → annotated_commit_from_ref → merge_analysis → (fast-forward | normal)` pipeline runs inside one `withLock`. No other task observes a mid-analysis state.
- **`cherrypick(_ commit:)`.** libgit2 internally touches MERGE_HEAD / CHERRY_PICK_HEAD, working tree, and the index. The Swift layer keeps the whole call under the lock.

### 4.3 Merge-analysis-driven dispatch in porcelain

`merge(_ branch:)` / `merge(branchNamed:)` dispatch on the `git_merge_analysis` result within the same critical section:

- `UP_TO_DATE` → no-op. Return `.upToDate`.
- `UNBORN` → `setHead(to: branch)` + `checkoutHead(options:)` using the caller-supplied `checkoutOptions`. Return `.unborn`. After `setHead`, HEAD points at the branch tip, so the subsequent `checkoutHead` populates the (previously empty) working tree from that tree.
- `FAST_FORWARD` → `setHead(to: branch)` + `checkoutHead(options:)` using the caller-supplied `checkoutOptions` (which defaults to empty `Strategy` == `GIT_CHECKOUT_SAFE`). Return `.fastForward`. (libgit2's `git_merge` itself does not fast-forward; the porcelain performs the standard dance described in `libgit2/examples/merge.c`.)
- `NORMAL` → `git_merge([ac], mergeOptions, checkoutOptions)`. libgit2 writes MERGE_HEAD / MERGE_MSG and merges into the index + working tree. Conflicts (if any) are surfaced via the index, not as Swift errors. Return `.normal`.

The returned `MergeAnalysis` OptionSet tells the caller which path ran.

### 4.4 Conflict model

libgit2 treats conflicts as a *normal intermediate state*, not an error — `git_merge` and `git_cherrypick` return 0 even when the resulting index has conflicts. v0.5a-i preserves that model:

- Default behavior: the call succeeds. Callers inspect `repository.state`, the resulting `Index.hasConflicts`, and `Index.conflicts()` (both added in v0.4b-i) to detect and resolve conflicts.
- Opt-in strict mode: set `MergeOptions.Flags.failOnConflict` (= `GIT_MERGE_FAIL_ON_CONFLICT`) to have libgit2 itself reject a conflicting merge. The Swift layer surfaces that as a normal `GitError` throw (`code == .mergeConflict`, empirical value to be confirmed during implementation).

### 4.5 Why `AnnotatedCommit` is explicit

libgit2's merge / analysis / rebase APIs take `git_annotated_commit *`, not raw commit OIDs, because the handle carries *provenance* — how the commit was resolved: from a local branch, from `FETCH_HEAD`, from a revspec. This provenance is what drives reflog messages and `ORIG_HEAD` recording. Hiding the handle would:

1. Lose reflog fidelity (the reflog would not know "merge of refs/heads/feature" — only "merge of <oid>").
2. Force us to invent internal helpers that pick a provenance source — the helper's guess is wrong for fetchhead-based flows (v0.5b) and rebase (v0.5a-ii).
3. Require re-introducing the handle later anyway: `git_rebase_init` takes three annotated commits and there is no overload that accepts raw commits.

Exposing `AnnotatedCommit` in v0.5a-i also clears the v0.4b-ii deferred `setHead(detachedAtAnnotated:)`.

### 4.6 Bare repository

libgit2's `git_merge` / `git_cherrypick` reject bare repositories via their internal working-tree checkout path; `mergeTrees` / `mergeCommits` / `mergeBase` / `mergeAnalysis` do not need a working tree and succeed on bare repos. v0.5a-i does not add a Swift-layer pre-check; libgit2's errors propagate unchanged. Matches the v0.4b-ii policy.

**Empirical note.** The observed error code / class for `git_merge` on a bare repo is **to be confirmed during implementation** and recorded in the v0.4b-ii-style "Empirical" block in the relevant test file.

### 4.7 Unborn HEAD

- `mergeAnalysis` returns `.unborn` in its analysis set when HEAD is unborn.
- Porcelain `merge(_ branch:)` handles `.unborn` per §4.3.
- Low-level `merge([ac])` on an unborn HEAD surfaces libgit2's error unchanged (documented as `code == .unbornBranch` pending empirical confirmation).
- `cherrypick` on an unborn HEAD surfaces libgit2's error unchanged.

### 4.8 Atomicity of porcelain merge

The porcelain `merge(_ branch:)` runs multiple libgit2 calls inside one lock (see §4.3). If `checkoutHead` fails after `setHead` succeeds during the fast-forward path, HEAD has moved but the working tree is stale — libgit2 does not roll back, and v0.5a-i does not add a Swift-layer rollback. Same trade-off as v0.4b-ii's `checkout(branch:)`. Tracked under §10.2.

### 4.9 `AnnotatedCommit` single-use convention

libgit2's `git_annotated_commit` is typically produced for a single merge / rebase / analysis call and then freed. Re-using the handle across multiple merges is not explicitly supported by libgit2's documentation. v0.5a-i documents this convention on the Swift `AnnotatedCommit` type but does not enforce it — the handle remains valid until Swift `deinit`, and libgit2 does not mutate the handle during merge calls. Callers who need deterministic behavior should create a fresh `AnnotatedCommit` per operation.

## 6. Types

### 6.1 `AnnotatedCommit`

```swift
// Sources/Git2/Objects/AnnotatedCommit.swift

import Cgit2

/// A libgit2 annotated commit.
///
/// An annotated commit carries both a commit OID and *provenance* — how the
/// commit was resolved (from a local ref, from FETCH_HEAD, from a revspec).
/// The provenance is what drives reflog messages and `ORIG_HEAD` recording
/// when the annotated commit is passed to `merge` / `setHead(detachedAtAnnotated:)`
/// / (in v0.5a-ii) `rebase`.
///
/// Instances are produced by ``Repository/annotatedCommit(for:)`` (reference or
/// OID) and ``Repository/annotatedCommit(from:)`` (commit).
///
/// An `AnnotatedCommit` is typically used for a single merge / analysis call
/// and then released. Re-using the same handle across multiple operations is
/// not explicitly supported by libgit2's documentation; callers who need
/// deterministic reflog behavior should create a fresh handle per operation.
public final class AnnotatedCommit: @unchecked Sendable {
    internal let handle: OpaquePointer
    internal let repository: Repository

    internal init(handle: OpaquePointer, repository: Repository) {
        self.handle = handle
        self.repository = repository
    }

    deinit {
        git_annotated_commit_free(handle)
    }

    /// The OID of the underlying commit.
    public var oid: OID { /* wraps git_annotated_commit_id */ }

    /// The ref name this annotated commit was resolved from, if any.
    /// Returns `nil` when the handle was created via OID lookup.
    public var refName: String? { /* wraps git_annotated_commit_ref; returns nil for empty */ }
}
```

### 6.2 `Repository.MergeOptions`

```swift
// Sources/Git2/Values/MergeOptions.swift

extension Repository {
    /// Options forwarded to libgit2's `git_merge`, `git_merge_trees`, and
    /// `git_merge_commits`.
    ///
    /// The default value (`MergeOptions()`) maps to libgit2's default — flags
    /// empty, file favor `.normal`, rename threshold 50%, target limit 200.
    public struct MergeOptions: Sendable, Equatable {
        public var flags: Flags
        public var fileFavor: FileFavor
        public var renameThreshold: Int
        public var targetLimit: Int

        public init(
            flags: Flags = [],
            fileFavor: FileFavor = .normal,
            renameThreshold: Int = 50,
            targetLimit: Int = 200
        )

        public struct Flags: OptionSet, Sendable, Equatable {
            public let rawValue: UInt32
            public init(rawValue: UInt32) { self.rawValue = rawValue }

            public static let findRenames    = Flags(rawValue: 1 << 0)  // GIT_MERGE_FIND_RENAMES
            public static let failOnConflict = Flags(rawValue: 1 << 1)  // GIT_MERGE_FAIL_ON_CONFLICT
            public static let skipReuc       = Flags(rawValue: 1 << 2)  // GIT_MERGE_SKIP_REUC
            public static let noRecursive    = Flags(rawValue: 1 << 3)  // GIT_MERGE_NO_RECURSIVE
            public static let virtualBase    = Flags(rawValue: 1 << 4)  // GIT_MERGE_VIRTUAL_BASE
        }

        public enum FileFavor: Sendable, Equatable {
            case normal    // GIT_MERGE_FILE_FAVOR_NORMAL
            case ours      // GIT_MERGE_FILE_FAVOR_OURS
            case theirs    // GIT_MERGE_FILE_FAVOR_THEIRS
            case union     // GIT_MERGE_FILE_FAVOR_UNION
        }
    }
}
```

Fields **not** surfaced: `metric` (function pointer), `recursion_limit`, `default_driver`, `file_flags`. Tracked under §10.2.

### 6.4 `Repository.MergeAnalysis`

```swift
extension Repository {
    /// Result of `git_merge_analysis`. Exposed as an `OptionSet` because
    /// libgit2 can return multiple bits simultaneously (e.g. `[.fastForward, .normal]`
    /// for a diverged-but-fast-forwardable history with a preference hint).
    public struct MergeAnalysis: OptionSet, Sendable, Equatable {
        public let rawValue: UInt32
        public init(rawValue: UInt32) { self.rawValue = rawValue }

        public static let none        = MergeAnalysis([])
        public static let normal      = MergeAnalysis(rawValue: 1 << 0)  // GIT_MERGE_ANALYSIS_NORMAL
        public static let upToDate    = MergeAnalysis(rawValue: 1 << 1)  // GIT_MERGE_ANALYSIS_UP_TO_DATE
        public static let fastForward = MergeAnalysis(rawValue: 1 << 2)  // GIT_MERGE_ANALYSIS_FASTFORWARD
        public static let unborn      = MergeAnalysis(rawValue: 1 << 3)  // GIT_MERGE_ANALYSIS_UNBORN
    }
}
```

### 7.3 Merge (stateful)

```swift
extension Repository {
    /// Low-level merge. Writes MERGE_HEAD / MERGE_MSG, updates the index,
    /// updates the working tree. Does **not** fast-forward — callers who
    /// want fast-forward on applicable histories should use the
    /// `Reference`-taking porcelain below, or analyze first.
    ///
    /// `heads.count` must equal 1 in v0.5a-i; octopus merge is deferred.
    public func merge(
        _ heads: [AnnotatedCommit],
        mergeOptions: MergeOptions = MergeOptions(),
        checkoutOptions: CheckoutOptions = CheckoutOptions()
    ) throws(GitError)

    /// Porcelain. Analyzes the requested merge, then dispatches:
    /// - `.upToDate` — no-op
    /// - `.unborn` — attach HEAD to the branch and checkout its tree
    /// - `.fastForward` — advance HEAD + safe checkout
    /// - `.normal` — call `git_merge` (writes MERGE_HEAD / MERGE_MSG,
    ///   conflicts land in the index)
    ///
    /// The returned `MergeAnalysis` reports which path ran.
    @discardableResult
    public func merge(
        _ branch: Reference,
        mergeOptions: MergeOptions = MergeOptions(),
        checkoutOptions: CheckoutOptions = CheckoutOptions()
    ) throws(GitError) -> MergeAnalysis

    /// Porcelain convenience: resolve `refs/heads/<name>` and forward to
    /// ``merge(_:mergeOptions:checkoutOptions:)-<Reference-overload>``.
    /// Local branches only (`GIT_BRANCH_LOCAL`).
    @discardableResult
    public func merge(
        branchNamed name: String,
        mergeOptions: MergeOptions = MergeOptions(),
        checkoutOptions: CheckoutOptions = CheckoutOptions()
    ) throws(GitError) -> MergeAnalysis
}
```

```swift
extension Repository {
    /// Current state (`git_repository_state`).
    public var state: State { get }

    /// Reads the repository's pending operation message (MERGE_MSG etc.).

    public func removeMessage() throws(GitError)

    /// Wraps `git_repository_state_cleanup`. Removes MERGE_HEAD / MERGE_MSG /
    /// CHERRY_PICK_HEAD / etc. Does **not** touch the working tree — callers
    /// who want a full "abort" should follow up with
    /// ``checkoutHead(options:)`` with ``CheckoutOptions/Strategy/force``.
    public func cleanupState() throws(GitError)
}
```

## 8. Concurrency / Locking

Unchanged from v0.4b-ii: every public entry point takes `repository.lock.withLock` once at the top.

The two multi-step operations call out in §4.2 (`merge(_ branch:)` porcelain and `cherrypick(_ commit:)`) run end-to-end inside a single critical section. No other task can observe the repository mid-analysis / mid-merge.

`AnnotatedCommit` is `@unchecked Sendable` following the existing handle convention. Its creation routes take the parent repository's lock; its `oid` and `refName` accessors are read-only on immutable libgit2 state and do not need the lock.

## 10. Open Questions / Deferrals

### 10.1 Open questions (to resolve during implementation)

- **Empirical error codes.** For each failure path that this spec lists without a confirmed `GitError.Code`/`.Class` — bare-repo merge, unborn-HEAD low-level merge, `failOnConflict` on conflict, unknown-branch porcelain, mainline-on-non-merge cherrypick — the actual observed value must be confirmed against libgit2 1.9.x during implementation and recorded in the relevant test file (following v0.4b-ii's "Empirical" block pattern).
- **`message()` / `removeMessage()` on a clean repo.** libgit2 returns `GIT_ENOTFOUND`. The spec claims `.notFound`; confirm during implementation.
- **OID array lifetime in `mergeBase(among:)`.** `git_merge_base_many` takes `const git_oid *input_array`; the bridge needs to keep the OID storage alive across the call. Pattern already in use via `withUnsafeBufferPointer`; verify no aliasing with `OID`'s internal storage.

### 10.2 Explicitly deferred (not landing in v0.5a-i)

Tracked under "Deferred from v0.5a-i" in TODO.md:

- **`git_merge_file` / `_from_index`.** File-level 3-way merge with configurable drivers.
- **`git_merge_bases` / `_bases_many` (plural).** Returning all common ancestors, not just one.
- **`git_merge_analysis_for_ref`.** Analyzing a merge into a non-HEAD ref.
- **`git_annotated_commit_from_fetchhead`.** Depends on fetchhead, which lands with v0.5b remote support.
- **`git_annotated_commit_from_revspec`.** Revspec is not yet surfaced anywhere in the public API.
- **`MergeOptions` deeper fields.** `metric` (function pointer), `recursion_limit`, `default_driver`, `file_flags`.
- **Octopus merge (`heads.count > 1`).** Signature already accepts the array; the runtime guard on `count == 1` is the only barrier.
- **Auto-commit porcelain.** `git merge` style automatic merge-commit creation on the non-conflicting `.normal` path.
- **`abortMerge()` / `abortCherrypick()` convenience.** Composable today via `cleanupState()` + `checkoutHead(strategy: .force)`.
- **Merge / cherry-pick callbacks (`notify_cb` / `progress_cb` / `perfdata_cb`).** Same deferral as v0.4b-ii's checkout callbacks.
- **`Commit` / `Reference`-accepting merge / cherry-pick overloads.** The explicit `AnnotatedCommit` path is the only way in v0.5a-i. Convenience overloads can land later additively.
- **Atomic rollback on porcelain merge.** If `checkoutHead` fails after `setHead` succeeds during fast-forward, the repo is left with HEAD moved but working tree stale. Same trade-off as v0.4b-ii's `checkout(branch:)`.
- **`AnnotatedCommit` re-use guarantees.** Currently documented as "single-use recommended". Formal re-use semantics deferred until a concrete need appears.

### 10.3 v0.5a-ii / v0.5b linkage

- v0.5a-ii (rebase) will consume `AnnotatedCommit` heavily (`git_rebase_init` takes three annotated commits). v0.5a-i's type is intended to be used as-is — no rebase-specific rework expected.
- v0.5b (remote / fetch / push) will add `git_annotated_commit_from_fetchhead` as an additional creation route on `Repository.annotatedCommit(...)`. Additive; no signature churn.
