# Git2 Swift Wrapper — v0.5c-ii Pull Porcelain

**Status:** Draft
**Date:** 2026-04-22
**Scope:** Second slice of the v0.5c "git CLI porcelain catch-up" line.
Adds `Repository.pull(remoteNamed:branchNamed:options:)` as a
merge-style pull porcelain composed from the existing
`fetch(remoteNamed:)` / `annotatedCommit(fromFetchHead:)` /
`merge(against:)` primitives. Builds on v0.5b-i
(`docs/superpowers/specs/2026-04-21-git2-v0.5b-i-network-fetch-design.md`),
v0.5a-i
(`docs/superpowers/specs/2026-04-20-git2-v0.5a-i-merge-cherrypick-design.md`),
and v0.5c-i
(`docs/superpowers/specs/2026-04-21-git2-v0.5c-i-repository-init-design.md`).

## 1. Purpose

`TODO.md` tracks two "Missing git CLI porcelain" items identified at
the end of the v0.5b-ii cycle:

1. **`Repository.create`** — landed in v0.5c-i.
2. **`Repository.pull`** — the subject of this slice.

Every ingredient for a merge-style pull porcelain already ships:

| Stage | Existing API (since) |
|---|---|
| Remote lookup | `Repository.lookupRemote(named:)` (v0.5b-i) |
| Network fetch | `Remote.fetch(refspecs:options:reflogMessage:)` (v0.5b-i) |
| `FETCH_HEAD`-based `AnnotatedCommit` | `Repository.annotatedCommit(fromFetchHead:remoteURL:oid:)` (v0.5b-i) |
| Tracking-ref lookup | `Repository.reference(named:)` (v0.2.0) |
| Merge analysis | `Repository.mergeAnalysis(against:)` (v0.5a-i) |
| Merge dispatch from `Reference` | `Repository.merge(_: Reference, …)` / `merge(branchNamed:, …)` (v0.5a-i) |

There is, however, a surface gap: **analysis-plus-dispatch entry points
exist for `Reference` and branch-name inputs, but not for
`AnnotatedCommit`.** Since the `FETCH_HEAD`-based
`AnnotatedCommit` is the canonical pull target (matching `git pull`
CLI reflog semantics), pull requires that missing entry. Closing this
gap is additionally useful independent of pull.

A second, smaller gap: **the wrapper has no way to read a local
branch's upstream tracking ref name** (`git_branch_upstream` /
`_name`). `Repository.pull` requires the caller to pass the branch
name explicitly, which is a thin but slightly bare experience when the
user has a tracking relationship already set. Exposing
`Reference.upstreamName()` as a sibling helper lets callers write
`try repo.head().upstreamName()` to derive the pull arguments in one
line without expanding the pull signature itself.

v0.5c-ii closes both gaps and composes them into the pull porcelain:

- **`Repository.pull(remoteNamed:branchNamed:options:) -> MergeAnalysis`** —
  merge-style pull. Fetches the specified branch from the specified
  remote via an explicit refspec, resolves the new tip from the
  updated remote-tracking ref, builds a `FETCH_HEAD`-provenance
  `AnnotatedCommit`, and runs analysis → dispatch inside a single
  critical section.
- **`Repository.merge(against: AnnotatedCommit, …) -> MergeAnalysis`** —
  analysis + dispatch entry point sibling to `merge(_: Reference, …)`.
  The existing `merge(_: Reference, …)` / `merge(branchNamed:, …)`
  implementations re-route through this new internal path; their
  external signatures, behavior, reflog strings, and error codes are
  unchanged.
- **`Reference.upstreamName() -> String?`** — thin wrapper over
  `git_branch_upstream_name`. Returns `nil` for non-branch refs and
  for local branches without a tracking relationship.

v0.5c-ii is **additive**: every v0.2–v0.5c-i public API keeps the
same signature and behavior.

## 2. Target Platforms

Unchanged from v0.5c-i: Apple platforms only, macOS 13+, iOS 16+,
Swift 6.2. No SSH, no SHA-256. `pull` reuses the transport backends
exposed by the XCFramework (HTTPS via libgit2's built-in HTTP + `file://`
for testing).

## 3. Scope

### 3.1 Included in v0.5c-ii

| Area | Additions |
|---|---|
| Repository API | `Repository.pull(remoteNamed:branchNamed:options:) -> MergeAnalysis` |
| Repository API | `Repository.merge(against: AnnotatedCommit, mergeOptions:, checkoutOptions:) -> MergeAnalysis` |

| Reference API | `Reference.upstreamName() -> String?` |
| Internal refactor | `performMerge` generalized to take `AnnotatedCommit` instead of `git_reference *`. `merge(_: Reference, …)` / `merge(branchNamed:, …)` internally build an `AnnotatedCommit` and delegate. |
| Test files | `Tests/Git2Tests/Remote/RepositoryPullTests.swift`, `Tests/Git2Tests/Merge/MergeAgainstAnnotatedTests.swift`, `Tests/Git2Tests/Reference/ReferenceUpstreamNameTests.swift` |

### 3.2 Explicitly not included in v0.5c-ii

All tracked in `TODO.md` under "Deferred from v0.5c-ii (pull porcelain)":

- **Rebase-style pull.** `PullOptions.strategy: .merge | .rebase` is
  not added. Rebase-style pull composes the `Rebase` step loop
  (`startRebase` → `next` → `commit` → `finish`); the natural building
  block is a `rebase(onto:)` auto-continue porcelain (deferred from
  v0.5a-ii) that does not yet exist. Landing rebase-style pull before
  that primitive would duplicate the step-loop logic and make later
  extraction a breaking change to `PullOptions`.
- **`pull(options:)` argument-less overload.** Convenience that
  resolves the remote and branch from HEAD's upstream config
  (`branch.<current>.remote` / `.merge`). `Reference.upstreamName()`
  already reduces this to a one-liner at the call site
  (`try repo.head().upstreamName()`); a first-class overload is
  additive and lands only if the pattern proves load-bearing.
- **Auto-commit on `.normal` merge.** Directly depends on v0.5a-i's
  "auto-commit porcelain" deferred item; the same single enabling
  change lifts both restrictions. Current behavior: `.normal` runs
  `git_merge`, writes `MERGE_HEAD` / `MERGE_MSG`, updates the index
  (with conflict entries if any), and returns — the caller commits.
- **Tag-control pull flag.** `git pull --tags` / `--no-tags` is
  satisfied by `options.fetch.downloadTags`. A pull-specific flag is
  redundant; revisit if a concrete use case wants tag control at the
  pull layer.
- **`FETCH_HEAD` line parsing API.** `annotatedCommit(fromFetchHead:…)`
  requires the caller to supply the OID. Pull bypasses
  `FETCH_HEAD`-file parsing by reading the updated remote-tracking
  ref directly (same OID after a just-completed fetch). Exposing a
  Swift value type for the full `FETCH_HEAD` file waits for a
  non-pull consumer.
- **HTTP / SSH-gated integration tests.** v0.5b-i / v0.5b-ii
  established an env-gated `RemoteGitHubIntegrationTests` pattern.
  Pull composes fetch + merge, both already covered over HTTP in
  v0.5b-i/b-ii; v0.5c-ii ships `file://` coverage only. An HTTP-path
  pull integration test is an additive future pass.
- **Pull-specific concurrency tests.** Pull is fetch + merge. Both
  already have concurrency tests (`RemoteConcurrencyTests`,
  `MergeConcurrencyTests`). Pull introduces no additional
  repository-side concurrency invariant — the fetch and merge
  phases take the lock independently, and the caller-visible
  contract is that the two phases are not atomic relative to
  other tasks on the same `Repository` handle (§7.3).
- **TSan on the pull path.** Blocked on the macOS 25.3 / Xcode 26.3
  `libclang_rt.tsan_osx_dynamic.dylib` code-signature rejection
  already tracked across v0.5b-i / b-ii.

### 3.3 Non-goals (unchanged)

- No shipped CLI.
- No 1:1 exposure of every libgit2 function.
- No Linux / Windows / Android.

## 4. Architectural Continuity

Every v0.2–v0.5c-i decision carries forward unchanged:

- **Thin-wrapper-over-libgit2.** `pull` is a composition; it adds no
  analysis, retry, or normalization beyond what the underlying
  primitives already do. `.normal` merge behavior, reflog message
  format, error codes, and conflict surface match
  `merge(_: Reference, …)` exactly.
- **Per-`Repository` lock.** `Remote.fetch` takes and releases the
  lock inside the network round-trip (unchanged from v0.5b-i).
  `pull` re-acquires the lock for the merge phase
  (tracking-ref lookup → annotated-commit build → analysis →
  dispatch), inside a single `lock.withLock { }` section. The two
  phases are **not** atomic relative to each other; this matches
  what callers see today when they call `fetch(remoteNamed:)`
  followed by `merge(branchNamed:)` directly. DocC documents this
  explicitly (§7.3).
- **Bootstrap precondition.** `Git.bootstrap()` must have been called.
  `pull` calls `GitRuntime.shared.requireBootstrapped()` at entry.
- **Typed throws.** `throws(GitError)` on every new API.
- **`Sendable`.** `PullOptions` is `Sendable`; closure fields inside
  `PullOptions.fetch` (credentials / certificate check / transfer
  progress) are already `@Sendable` per v0.5b-i's `FetchOptions`.
- **Error code specificity.** Test assertions follow the
  `code == .specific` / `class == .specific` pattern established in
  v0.3 / v0.4 / v0.5a-i / b-i / b-ii.

## 5. Public API

### 5.1 `Repository.pull`

```swift
extension Repository {
    /// Fetches from a remote and merges the fetched branch into the
    /// current HEAD. Thin porcelain composed of ``fetch(remoteNamed:refspecs:options:reflogMessage:)``,
    /// ``annotatedCommit(fromFetchHead:remoteURL:oid:)``, and
    /// ``merge(against:mergeOptions:checkoutOptions:)``.
    ///
    /// Pull fetches a single branch from `remoteName` using an explicit
    /// refspec `refs/heads/<branchName>:refs/remotes/<remoteName>/<branchName>`
    /// so that the post-fetch state has a predictable single-branch
    /// `FETCH_HEAD` and a corresponding remote-tracking ref. The merge
    /// phase reads the remote-tracking ref for the OID and builds the
    /// `AnnotatedCommit` via
    /// ``annotatedCommit(fromFetchHead:remoteURL:oid:)`` so that the
    /// reflog entry records the merge as originating from `FETCH_HEAD`
    /// — matching `git pull` CLI semantics.
    ///
    /// - Parameters:
    ///   - remoteName: Remote configured on this repository (e.g.
    ///     `"origin"`). Must resolve via ``lookupRemote(named:)``.
    ///   - branchName: Branch on the remote to pull
    ///     (e.g. `"main"`). No `refs/heads/` prefix.
    ///   - options: See ``PullOptions``.
    /// - Returns: The ``MergeAnalysis`` result of the merge phase.
    ///   `.upToDate` / `.fastForward` / `.unborn` / `.normal` are
    ///   possible; see ``merge(against:mergeOptions:checkoutOptions:)``
    ///   for the per-case behavior.
    /// - Throws: ``GitError``.
    ///   - Fetch-phase errors propagate unchanged from
    ///     ``Remote/fetch(refspecs:options:reflogMessage:)``
    ///     (auth / network / HTTP / credentials-callback).
    ///   - ``GitError/Code/notFound`` / ``GitError/Class/reference``
    ///     when `refs/remotes/<remoteName>/<branchName>` is absent
    ///     after the fetch (e.g. the branch does not exist on the
    ///     remote).
    ///   - ``GitError/Code/nonFastForward`` /
    ///     ``GitError/Class/merge`` when
    ///     `options.allowNonFastForward == false` and the analysis
    ///     returns `.normal`. The fetch has already completed; HEAD is
    ///     unchanged.
    ///   - Merge-phase errors (checkout conflicts on dirty working
    ///     tree, bare-repo rejection) propagate from
    ///     ``merge(against:mergeOptions:checkoutOptions:)``.
    ///
    /// - Important: This porcelain is **not atomic** across the fetch
    ///   and merge phases. The per-repository lock is released between
    ///   the two phases. Another task operating on the same
    ///   ``Repository`` handle may observe the repository in a
    ///   "fetched but not yet merged" state. Callers requiring a
    ///   stronger guarantee should serialize at the call site.
    public func pull(
        remoteNamed remoteName: String,
        branchNamed branchName: String,
        options: PullOptions = PullOptions()
    ) throws(GitError) -> MergeAnalysis
}
```

### 5.2 `Repository.PullOptions`

```swift
extension Repository {
    /// Configuration for ``pull(remoteNamed:branchNamed:options:)``.
    public struct PullOptions: Sendable {
        /// Fetch-phase options (credentials / certificate check /
        /// transfer progress / prune / depth / etc). See
        /// ``FetchOptions``.
        public var fetch: FetchOptions

        /// Merge-phase options. Forwarded verbatim to
        /// ``merge(against:mergeOptions:checkoutOptions:)``.
        public var merge: MergeOptions

        /// Checkout-phase options applied when the analysis resolves
        /// to `.fastForward`, `.unborn`, or `.normal`. Forwarded
        /// verbatim to the merge path.
        public var checkout: CheckoutOptions

        /// When `false`, the pull aborts with
        /// ``GitError/Code/nonFastForward`` / ``GitError/Class/merge``
        /// if the merge analysis returns `.normal` (i.e. a real
        /// merge commit would be required). The fetch phase has
        /// already completed at that point; the remote-tracking ref
        /// and `FETCH_HEAD` are updated but HEAD and the working tree
        /// are untouched. `.fastForward`, `.upToDate`, and `.unborn`
        /// are unaffected. Default `true`.
        public var allowNonFastForward: Bool

        /// Reflog message written by the fetch phase on the
        /// remote-tracking ref and `FETCH_HEAD`. `nil` delegates to
        /// libgit2's default (`"fetch"`). Note: the merge phase's
        /// reflog message is libgit2's automatic
        /// `"merge FETCH_HEAD: …"` string and is not controllable
        /// through this option.
        public var reflogMessage: String?

        public init(
            fetch: FetchOptions = FetchOptions(),
            merge: MergeOptions = MergeOptions(),
            checkout: CheckoutOptions = CheckoutOptions(),
            allowNonFastForward: Bool = true,
            reflogMessage: String? = nil
        )
    }
}
```

### 5.3 `Repository.merge(against:)`

```swift
extension Repository {
    /// Analyzes and dispatches a merge against a pre-built
    /// ``AnnotatedCommit``. Sibling to
    /// ``merge(_:mergeOptions:checkoutOptions:)-Reference`` and
    /// ``merge(branchNamed:mergeOptions:checkoutOptions:)``; closes
    /// the surface gap where ``AnnotatedCommit`` was the only merge
    /// input that could not drive analysis + dispatch.
    ///
    /// The entire analysis → dispatch sequence runs inside a single
    /// critical section — no other task observes a mid-merge state.
    ///
    /// - Parameters:
    ///   - annotated: The merge target. Typically produced by
    ///     ``annotatedCommit(for:)-Reference``,
    ///     ``annotatedCommit(for:)-OID``,
    ///     ``annotatedCommit(from:)``, or
    ///     ``annotatedCommit(fromFetchHead:remoteURL:oid:)``.
    ///   - mergeOptions: Merge rename / favor / recursion options.
    ///   - checkoutOptions: Checkout options applied to
    ///     `.fastForward`, `.unborn`, and `.normal` dispatch paths.
    /// - Returns: The ``MergeAnalysis`` bits libgit2 produced
    ///   (prior to dispatch). `.preference` is intentionally dropped
    ///   to match ``merge(_:mergeOptions:checkoutOptions:)-Reference``.
    /// - Throws: ``GitError`` from `git_merge_analysis`,
    ///   `git_checkout_tree` / `_head`, `git_repository_set_head`,
    ///   or `git_merge`. Conflicts in the `.normal` path do **not**
    ///   throw; inspect `repo.state == .merge` and
    ///   `repo.index().hasConflicts` after return.
    public func merge(
        against annotated: AnnotatedCommit,
        mergeOptions: MergeOptions = MergeOptions(),
        checkoutOptions: CheckoutOptions = CheckoutOptions()
    ) throws(GitError) -> MergeAnalysis
}
```

### 5.4 `Reference.upstreamName()`

```swift
extension Reference {
    /// Resolves this local branch's upstream tracking ref name
    /// (e.g. `"refs/remotes/origin/main"`). Thin wrapper over
    /// `git_branch_upstream_name`.
    ///
    /// Returns `nil` when:
    /// - the ref is not a local branch (remote-tracking refs, tags,
    ///   symbolic HEAD when detached),
    /// - the local branch has no `branch.<name>.remote` /
    ///   `branch.<name>.merge` configuration,
    /// - libgit2 returns `GIT_ENOTFOUND` for any other reason.
    ///
    /// Any other libgit2 error (e.g. a broken config file) propagates
    /// as ``GitError``.
    public func upstreamName() throws(GitError) -> String?
}
```

## 6. Implementation

### 6.1 `performMerge` refactor

Today `performMerge(branchHandle:mergeOptions:checkoutOptions:)` takes
a `git_reference *`, internally builds an `AnnotatedCommit` via
`git_annotated_commit_from_ref`, and runs analysis → dispatch. The
refactor:

```swift
// Before
internal func performMerge(
    branchHandle: OpaquePointer,   // git_reference *
    mergeOptions: MergeOptions,
    checkoutOptions: CheckoutOptions
) throws(GitError) -> MergeAnalysis

// After
internal func performMerge(
    annotated: AnnotatedCommit,
    mergeOptions: MergeOptions,
    checkoutOptions: CheckoutOptions
) throws(GitError) -> MergeAnalysis
```

Callers updated:

- `merge(_: Reference, mergeOptions:, checkoutOptions:)` now builds an
  `AnnotatedCommit` via `annotatedCommitLocked(for: reference)` (the
  no-lock internal sibling of `annotatedCommit(for:)`) before calling
  `performMerge`.
- `merge(branchNamed:, …)` unchanged at the entry — still resolves the
  branch ref, then calls `merge(_: Reference, …)` via the existing
  path.
- `merge(against:, …)` (new) is a thin wrapper: enters
  `lock.withLock`, calls `performMerge(annotated:, …)`.

Reflog messages: libgit2's `git_annotated_commit_from_ref` records the
branch name as provenance; `git_annotated_commit_from_fetchhead`
records `FETCH_HEAD` with the branch/URL. Because both paths now
produce an `AnnotatedCommit` before dispatch, reflog output is
unchanged per input: `merge(_: Reference)` still writes
`"merge refs/heads/<name>: …"` and `pull` writes
`"merge FETCH_HEAD: …"`.

### 6.2 `pull` implementation

```swift
public func pull(
    remoteNamed remoteName: String,
    branchNamed branchName: String,
    options: PullOptions = PullOptions()
) throws(GitError) -> MergeAnalysis {
    try GitRuntime.shared.requireBootstrapped()

    // --- Phase 1: fetch (lock held inside Remote.fetch) ---
    let refspec = try Refspec(
        "refs/heads/\(branchName):refs/remotes/\(remoteName)/\(branchName)"
    )
    let remote = try lookupRemote(named: remoteName)
    try remote.fetch(
        refspecs: [refspec],
        options: options.fetch,
        reflogMessage: options.reflogMessage
    )
    let remoteURL = remote.url ?? ""

    // --- Phase 2: merge (new lock section) ---
    return try lock.withLock { () throws(GitError) -> MergeAnalysis in
        let trackingRefName = "refs/remotes/\(remoteName)/\(branchName)"
        guard let trackingRef = try referenceLocked(named: trackingRefName)
        else {
            throw GitError(
                code: .notFound, class: .reference,
                message: "no tracking ref \(trackingRefName) after fetch"
            )
        }
        let oid = try trackingRef.targetLocked()

        let annotated = try annotatedCommitLocked(
            fromFetchHead: branchName,
            remoteURL: remoteURL,
            oid: oid
        )

        if !options.allowNonFastForward {
            let (analysis, _) = try mergeAnalysisLocked(against: [annotated])
            let ffOrTrivial =
                analysis.contains(.fastForward)
                || analysis.contains(.upToDate)
                || analysis.contains(.unborn)
            if analysis.contains(.normal) && !ffOrTrivial {
                throw GitError(
                    code: .nonFastForward, class: .merge,
                    message: "pull would require non-fast-forward merge"
                )
            }
        }

        return try performMerge(
            annotated: annotated,
            mergeOptions: options.merge,
            checkoutOptions: options.checkout
        )
    }
}
```

Helper functions introduced as `internal` no-lock siblings where
missing (following the v0.5a-i pattern):

- `referenceLocked(named:) -> Reference?` — no-lock sibling of
  `reference(named:)`.
- `Reference.targetLocked() -> OID` — no-lock sibling of
  `Reference.target`.
- `annotatedCommitLocked(for: Reference) -> AnnotatedCommit` —
  no-lock sibling of `annotatedCommit(for: Reference)`. Consumed by
  the refactored `merge(_: Reference, …)` path (§6.1) so the whole
  ref → annotated → dispatch sequence stays under one lock.
- `annotatedCommitLocked(fromFetchHead:remoteURL:oid:) -> AnnotatedCommit` —
  no-lock sibling of `annotatedCommit(fromFetchHead:…)`. Consumed by
  `pull` (§6.2).
- `mergeAnalysisLocked(against:) -> (MergeAnalysis, MergePreference)` —
  no-lock sibling of `mergeAnalysis(against:)`.

Each is a mechanical extraction: take the body of the public method,
remove the `lock.withLock { }` wrapper.

### 6.3 `Reference.upstreamName`

```swift
public func upstreamName() throws(GitError) -> String? {
    try repository.lock.withLock { () throws(GitError) -> String? in
        var buf = git_buf()
        defer { git_buf_dispose(&buf) }
        let rc = git_branch_upstream_name(
            &buf,
            repository.handle,
            name
        )
        if rc == GIT_ENOTFOUND.rawValue { return nil }
        try check(rc)
        guard let ptr = buf.ptr else { return nil }
        return String(cString: ptr)
    }
}
```

The `GIT_ENOTFOUND` → `nil` mapping mirrors
`Repository.reference(named:)` and matches the documented behavior of
`git_branch_upstream_name` (returns `GIT_ENOTFOUND` for non-branches
and for branches without tracking configuration).

### 7.1 `Tests/Git2Tests/Remote/RepositoryPullTests.swift`

- `pull_upToDate_returnsUpToDate` — upstream and downstream at the
  same OID. Assert result `.contains(.upToDate)`, HEAD OID unchanged,
  no `MERGE_HEAD`.
- `pull_fastForward_movesHead` — upstream advanced by one commit,
  downstream at old tip. Assert result `.contains(.fastForward)`,
  HEAD OID equals upstream tip, working tree reflects new content,
  no `MERGE_HEAD`.

## 8. Errors

### 8.1 `Repository.pull`

| Trigger | `GitError` shape | Source |
|---|---|---|
| `Git2.bootstrap()` not called | `.user` / `.invalid` / message | `GitRuntime.requireBootstrapped` (existing contract) |
