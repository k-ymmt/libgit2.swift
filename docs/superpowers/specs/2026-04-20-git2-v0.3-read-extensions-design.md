# Git2 Swift Wrapper — v0.3.0 Read Extensions

**Status:** Draft
**Date:** 2026-04-20
**Scope:** Second slice of the Git2 Swift wrapper. Builds on the v0.2.0 foundation (`docs/superpowers/specs/2026-04-20-git2-swift-wrapper-foundation-design.md`) and adds a full read-only API surface: repository discovery, reference iteration/lookup, the full object family (`Tree`, `Blob`, `Tag`, polymorphic `Object`), tree-to-tree diff, sortable revwalks, and a public `RevWalk` type.

## 1. Purpose

`Git2` v0.2.0 shipped the minimal first slice — `Repository.open` → `head` → `log`. That is enough to read commit metadata but leaves out every other read operation users typically need (inspecting a tree, reading a blob's content, reading annotated tags, listing branches and tags, computing diffs between revisions).

v0.3.0 completes the read-only surface. Write operations (commit/branch/tag creation, index, HEAD manipulation) remain out of scope and are planned for v0.4.

v0.3.0 is **additive**: every existing v0.2.0 public API keeps the same signature and behavior. Downstream code that worked against v0.2.0 keeps working.

## 2. Target Platforms

Unchanged from v0.2.0: Apple platforms only, macOS 13+, iOS 16+, Swift 6.2.

## 3. Scope

### 3.1 Included in v0.3.0

| Area | Additions |
|---|---|
| Values | `ObjectKind`, `TreeEntry`, `TreeEntry.FileMode` |
| Objects | `Object` enum, `Tree`, `Blob`, `Tag` (annotated) |
| Repository | `discover(startingAt:acrossFilesystems:ceilingDirectories:)`, `open(discoveringFrom:...)`, `object(for:)`, `reference(named:)`, `references()`, `diff(from:to:)` |
| Log / Walk | Public `RevWalk` class, `CommitSequence.Sorting` OptionSet, `log(from:sorting:)` overload |
| Reference | (no new API — the annotated-tag peel path is already handled by `resolveToCommit()`; v0.3 only adds test coverage) |
| Diff | `Diff`, `DiffDelta`, `DiffFile`, `DiffDelta.Status` |
| Internal | `ObjectHandle` helper; migration of `OID(hex:)` from the deprecated `git_oid_fromstr` to `git_oid_fromstrn` |

### 3.2 Explicitly not included in v0.3.0

- `DiffOptions` (rename / copy detection, context lines)
- Hunk- and line-level diff (`DiffHunk`, `DiffLine`)
- Working-directory / index diffs (`diff(tree:toWorkdir:)`, `diff(tree:toIndex:)`, `diff(indexToWorkdir:)`). Deferred to v0.4 alongside the index / staging API
- Recursive tree walk helper (`tree.walk { ... }`)
- Commit / branch / tag creation (write operations)
- `Reference.type` (direct vs. symbolic) / public `Reference.resolve()` — current `target` / `resolveToCommit()` cover the read cases
- SSH, remote / fetch / push, merge / rebase / cherry-pick, index / staging
- SHA-256 repositories (requires `EXPERIMENTAL_SHA256` in libgit2)

### 3.3 Non-goals (unchanged from v0.2.0)

- No shipped CLI.
- No 1:1 exposure of every libgit2 function; low-level access remains via `@_exported import Cgit2`.
- No Linux / Windows / Android support.
- No Keychain / credential-UI integration.

## 4. Architectural Continuity With v0.2.0

Every v0.2.0 foundational decision carries forward unchanged:

- **Hybrid resource ownership.** Handles (`Repository`, `Reference`, `Commit`, and the new `Tree` / `Blob` / `Tag` / `Diff` / `RevWalk`) are `final class: @unchecked Sendable`. Snapshot values (`OID`, `Signature`, `Version`, `GitError`, and the new `TreeEntry`, `DiffDelta`, `DiffFile`) are structs.
- **Parent-child lifetime.** Every object handle holds a strong reference to its owning `Repository`. `TreeEntry` / `DiffDelta` / `DiffFile` are snapshot values and have no such tie.
- **Typed throws.** Every failing API uses `throws(GitError)`.
- **Per-`Repository` serialization.** All libgit2 calls that touch a repo or any of its children are executed inside `repository.lock.withLock`. New types follow the same rule.
- **Error representation.** Still `GitError` with `Code` / `Class` enums; no new error types introduced.
- **Lifecycle.** Still `Git.bootstrap()` / `Git.shutdown()`; opening a repository without bootstrap still trips a `preconditionFailure`.

## 6. Internal Infrastructure

### 6.1 `ObjectHandle`

`Tree`, `Blob`, `Tag` all wrap `git_object *` (`git_object_lookup` returns an opaque type-discriminated handle). Their common lookup and peel operations land in one helper so the public types stay thin.

```swift
// Sources/Git2/Internal/ObjectHandle.swift
import Cgit2

internal enum ObjectHandle {
    /// Looks up an object. `kind == GIT_OBJECT_ANY` matches any object type and
    /// lets the caller read back the actual kind via `git_object_type`.
    /// Caller owns the returned handle and must free it with `git_object_free`.
    /// Caller holds `repository.lock` across this call.
    static func lookup(
        repository: Repository,
        oid: OID,
        kind: git_object_t
    ) throws(GitError) -> OpaquePointer

    /// Peels a tag to its ultimate non-tag target. Caller owns the returned handle.
    /// Caller holds `repository.lock` across this call.
    static func peelToKind(
        repository: Repository,
        handle: OpaquePointer,
        kind: git_object_t
    ) throws(GitError) -> OpaquePointer
}
```

Lock is taken by the caller so that multi-step operations (lookup → inspect → peel) stay inside a single critical section.

### 6.2 `git_oid_fromstr` migration

`OID(hex:)` currently calls `git_oid_fromstr`, deprecated in libgit2 1.x. v0.3 switches to `git_oid_fromstrn(&oid, cstr, hex.utf8.count)`. The signature, behavior, and tests of `OID(hex:)` do not change.

## 7. Object Family

### 7.1 `ObjectKind`

```swift
// Sources/Git2/Objects/ObjectKind.swift

/// The kind of a Git object in the object database.
public enum ObjectKind: Sendable, Equatable {
    case commit
    case tree
    case blob
    case tag
}

extension ObjectKind {
    internal static func from(_ raw: git_object_t) -> ObjectKind?
    internal var raw: git_object_t { get }
}
```

Unknown values from libgit2 (e.g. the reserved `GIT_OBJECT_OFS_DELTA` / `GIT_OBJECT_REF_DELTA`) are mapped to `nil` in `from(_:)`. These only appear during packfile I/O, not in user-level lookups, so they never reach the public API.

### 7.2 `Object` enum

```swift
// Sources/Git2/Objects/Object.swift

/// A polymorphic reference to a single object in the repository's object database.
///
/// Obtained via ``Repository/object(for:)``.  Each case owns a handle to the
/// concrete object type (`Commit`, `Tree`, `Blob`, or `Tag`).
public enum Object: Sendable {
    case commit(Commit)
    case tree(Tree)
    case blob(Blob)
    case tag(Tag)

    /// The object's kind (constant-time — does not read the ODB).
    public var kind: ObjectKind { get }

    /// The object's OID (constant-time).
    public var oid: OID { get }
}
```

`Object` is a value type; iterating or switching over it is free. The concrete cases (`Commit` / `Tree` / `Blob` / `Tag`) remain class handles, so the enum case payload is a single reference.

### 7.3 `Repository.object(for:)`

```swift
// Sources/Git2/Repository/Repository+Objects.swift

extension Repository {
    /// Looks up an object of any kind.
    ///
    /// - Returns: The object wrapped in the appropriate ``Object`` case, or `nil`
    ///   if no object exists with the given OID.
    /// - Throws: ``GitError`` only for genuine failures (I/O error, corruption).
    public func object(for oid: OID) throws(GitError) -> Object?
}
```

"Not found" is Optional rather than `.notFound` throw (per Q9: existence checks read better as `if let`). Same pattern as `reference(named:)`.

### 7.4 `Tree`

```swift
// Sources/Git2/Objects/Tree.swift

public final class Tree: @unchecked Sendable {
    public let repository: Repository
    internal let handle: OpaquePointer

    internal init(handle: OpaquePointer, repository: Repository)
    deinit   // git_tree_free

    public var oid: OID { get }

    /// The number of entries in this tree (direct children only — not recursive).
    public var count: Int { get }

    /// An entry by index. Traps on out-of-range access, consistent with `Array`.
    public subscript(index: Int) -> TreeEntry { get }

    /// An entry by name, or `nil` if no entry with that name exists.
    public subscript(name name: String) -> TreeEntry? { get }
}
```

v0.3 does **not** add `Sequence` / `Collection` conformance (per Q7). Callers iterate via `(0..<tree.count).map { tree[$0] }` or a `for i in 0..<tree.count` loop. Sub-trees are looked up by OID: `try repository.object(for: entry.oid)` → `case .tree(let sub) = obj`. No recursive walk helper.

### 7.5 `TreeEntry`

```swift
// Sources/Git2/Objects/Tree.swift (same file)

public struct TreeEntry: Sendable, Equatable {
    public let name: String
    public let oid: OID
    public let kind: ObjectKind
    public let filemode: FileMode

    public enum FileMode: Sendable, Equatable {
        case tree
        case blob
        case blobExecutable
        case link
        case commit    // submodule
    }
}
```

`TreeEntry` is a **snapshot value**: the init site copies `name` / `oid` / `kind` / `filemode` out of the borrowed `git_tree_entry *` returned by libgit2 (which is only valid while the parent tree is alive). After construction the value is independent of any libgit2 handle, which is what makes `Sendable` and `Equatable` honest.

`FileMode.from(git_filemode_t)` is an internal helper. `GIT_FILEMODE_UNREADABLE` (which libgit2 uses as a sentinel) is not expected in a read of a valid tree; `from(_:)` traps on it rather than synthesizing a fallback case.

### 7.6 `Blob`

```swift
// Sources/Git2/Objects/Blob.swift

public final class Blob: @unchecked Sendable {
    public let repository: Repository
    internal let handle: OpaquePointer

    internal init(handle: OpaquePointer, repository: Repository)
    deinit   // git_blob_free

    public var oid: OID { get }

    /// The raw size of the blob in bytes.
    public var size: Int64 { get }

    /// The blob's raw content, copied into Swift-owned storage.
    public var content: Data { get }

    /// libgit2's heuristic guess at whether the content is binary (NUL bytes in
    /// the first N bytes). Not a certainty — it's the same heuristic Git uses
    /// to decide whether to apply textual diffs.
    public var isBinary: Bool { get }
}
```

`content` returns a full copy. Streamed access is deferred to a later slice if a concrete use case needs it.

### 7.7 `Tag` (annotated tag only)

```swift
// Sources/Git2/Objects/Tag.swift

/// An annotated Git tag — a full object in the database with its own OID,
/// tagger signature, message, and target.
///
/// Lightweight tags (refs in `refs/tags/` that point directly at a commit) are
/// not ``Tag`` values; they are represented by ``Reference``.
public final class Tag: @unchecked Sendable {
    public let repository: Repository
    internal let handle: OpaquePointer

    internal init(handle: OpaquePointer, repository: Repository)
    deinit   // git_tag_free

    public var oid: OID { get }
    public var name: String { get }
    public var message: String { get }

    /// The tagger signature. `nil` is theoretically possible for some exotic
    /// tags; libgit2 treats it as optional.
    public var tagger: Signature? { get }

    public var targetOID: OID { get }
    public var targetKind: ObjectKind { get }

    /// Resolves the tag's target. For tag-of-tag chains, call `target()` on the
    /// returned ``Object`` case recursively; or use
    /// ``Reference/resolveToCommit()`` if you just need the terminal commit.
    public func target() throws(GitError) -> Object
}
```

## 8. RevWalk, Sorting, and Log Rewiring

### 8.1 Public `RevWalk`

```swift
// Sources/Git2/Walker/RevWalk.swift

/// Commit graph traversal with full error reporting and sort control.
///
/// Unlike ``CommitSequence``, `RevWalk.next()` throws on iteration errors,
/// so callers that need to distinguish normal termination from mid-walk
/// failure use this type directly.
public final class RevWalk: @unchecked Sendable {
    public let repository: Repository
    internal let handle: OpaquePointer

    /// Creates a new walker. A freshly created walker has no push points and
    /// returns `nil` from `next()` until at least one push* call.
    public init(repository: Repository) throws(GitError)
    deinit   // git_revwalk_free

    // Push: start walking from these roots.
    public func push(_ commit: Commit) throws(GitError)
    public func push(oid: OID) throws(GitError)
    public func push(refName: String) throws(GitError)
    public func pushHead() throws(GitError)

    // Hide: prune the subgraph reachable from these commits.
    public func hide(_ commit: Commit) throws(GitError)
    public func hide(oid: OID) throws(GitError)

    // Options.
    public func setSorting(_ sorting: CommitSequence.Sorting) throws(GitError)
    public func simplifyFirstParent() throws(GitError)

    /// Resets the walker to its initial state (clears push / hide / sorting).
    public func reset()

    /// The next commit in the walk.
    ///
    /// - Returns: The next commit, or `nil` when the walk terminates normally
    ///   (libgit2's `GIT_ITEROVER`).
    /// - Throws: ``GitError`` on mid-walk failure (ODB error, corruption, etc.).
    public func next() throws(GitError) -> Commit?
}
```

Not `Sequence` / `IteratorProtocol` — the protocol can't forward `throws`, and the whole point of this type is to surface errors. Callers write:

```swift
let walk = try RevWalk(repository: repo)
try walk.push(tip)
while let commit = try walk.next() {
    // ...
}
```

### 8.2 `CommitSequence.Sorting`

```swift
// Sources/Git2/Walker/RevWalk.swift (same file, extension)

extension CommitSequence {
    public struct Sorting: OptionSet, Sendable {
        public let rawValue: UInt32
        public init(rawValue: UInt32)

        public static let topological = Sorting(rawValue: UInt32(GIT_SORT_TOPOLOGICAL.rawValue))
        public static let time        = Sorting(rawValue: UInt32(GIT_SORT_TIME.rawValue))
        public static let reverse     = Sorting(rawValue: UInt32(GIT_SORT_REVERSE.rawValue))

        /// Insertion order (libgit2 default). `[]` and `.none` are equivalent.
        public static let none: Sorting = []
    }
}
```

`Sorting` nests under `CommitSequence` (not `RevWalk`) because both types consume it and `CommitSequence` is the older public type from v0.2. Follows Foundation's convention (e.g. `Calendar.Component`).

### 8.3 `log` API

```swift
// Sources/Git2/Repository/Repository+Log.swift

extension Repository {
    // v0.2, unchanged signature and behavior:
    public func log(from start: Commit) -> CommitSequence

    // v0.3 addition:
    public func log(from start: Commit, sorting: CommitSequence.Sorting) -> CommitSequence
}
```

Internal: both overloads produce the same `CommitSequence` struct, which gains a `sorting` field (defaults to `.none` for the old overload). `CommitSequence.makeIterator()` is rewritten to:

```swift
public func makeIterator() -> CommitIterator {
    CommitIterator(repository: repository, startOID: startOID, sorting: sorting)
}
```

`CommitIterator` now owns a `RevWalk` internally, sets sorting, pushes the start OID, and calls `walk.next()` in `next()` — discarding errors (silent-fail) to preserve the `IteratorProtocol` contract. That is the deliberate tradeoff: `CommitSequence` stays `for-in`-friendly for casual use, and callers who need error detection switch to `RevWalk` directly.

The internal `RevWalkHandle` (`Sources/Git2/Internal/RevWalkHandle.swift`) is deleted; its callers now go through `RevWalk`.

// Sources/Git2/References/ReferenceSequence.swift

public struct ReferenceSequence: Sequence {
    public typealias Element = Reference

    internal let repository: Repository

    public func makeIterator() -> ReferenceIterator
}

public struct ReferenceIterator: IteratorProtocol {
    public typealias Element = Reference

    /// Returns the next reference, or `nil` when iteration completes or fails.
    /// Mid-iteration errors are swallowed to preserve the `IteratorProtocol`

## 10. Diff (tree-to-tree, file-level)

### 10.1 Entry point

```swift
// Sources/Git2/Repository/Repository+Diff.swift

extension Repository {
    /// Computes the file-level diff between two trees.
    ///
    /// - Parameters:
    ///   - old: The "from" tree, or `nil` for the empty tree (useful for the
    ///     initial commit, where every file is an addition).
    ///   - new: The "to" tree, or `nil` for the empty tree (useful for
    ///     computing a full deletion set).
    /// - Returns: A ``Diff`` describing file-level changes. Hunk and line
    ///   granularity are deferred to a later slice.
    public func diff(from old: Tree?, to new: Tree?) throws(GitError) -> Diff
}
```

At least one of `old` / `new` must be non-nil. Both nil throws `GitError(code: .invalid, class: .invalid, message: "diff requires at least one tree")` without calling into libgit2.

### 10.2 `Diff`

```swift
// Sources/Git2/Diff/Diff.swift

public final class Diff: @unchecked Sendable {
    public let repository: Repository
    internal let handle: OpaquePointer

    internal init(handle: OpaquePointer, repository: Repository)
    deinit   // git_diff_free

    public var count: Int { get }                       // git_diff_num_deltas
    public subscript(index: Int) -> DiffDelta { get }   // git_diff_get_delta → snapshot
}
```

`Diff` is a handle (class). Indexing copies the `git_diff_delta` pointee into a Swift value so the returned `DiffDelta` is independent of the `Diff`'s lifetime.

### 10.3 `DiffDelta` and `DiffFile`

```swift
// Sources/Git2/Diff/DiffDelta.swift

public struct DiffDelta: Sendable, Equatable {
    public let status: Status
    public let oldFile: DiffFile
    public let newFile: DiffFile

    public enum Status: Sendable, Equatable {
        case unmodified
        case added
        case deleted
        case modified
        case renamed
        case copied
        case ignored
        case untracked
        case typeChange
        case unreadable
        case conflicted
    }
}

// Sources/Git2/Diff/DiffFile.swift

public struct DiffFile: Sendable, Equatable {
    public let oid: OID
    public let path: String
    public let size: UInt64
    public let mode: TreeEntry.FileMode
}
```

`oldFile` and `newFile` are non-optional. libgit2's convention for one-sided deltas is to set the absent side's OID to all-zero; `DiffFile.oid == OID.zero` (where `OID.zero` is the all-zero OID) reports "no file on this side" without needing an extra optional layer.

An `OID.zero` static is added as part of v0.3 (trivial — a zero-initialized `git_oid`).

## 11. Discovery

```swift
// Sources/Git2/Repository/Repository+Discover.swift

extension Repository {
    /// Finds the Git directory of a repository by walking up from `url`.
    ///
    /// - Parameters:
    ///   - url: The starting directory (or a file inside the working tree).
    ///   - acrossFilesystems: If `false` (default), the search stops when it
    ///     would cross a filesystem mount point. If `true`, it keeps going.
    ///   - ceilingDirectories: Directories at which to stop searching. The
    ///     search halts as soon as it reaches any of these or walks past them.
    /// - Returns: The URL of the discovered `.git` directory (or the bare
    ///   repository root).
    /// - Throws: ``GitError/Code/notFound`` if no repository is found before
    ///   hitting a ceiling or filesystem boundary.
    public static func discover(
        startingAt url: URL,
        acrossFilesystems: Bool = false,
        ceilingDirectories: [URL] = []
    ) throws(GitError) -> URL

    /// Convenience: discovers and opens a repository in one call.
    public static func open(
        discoveringFrom url: URL,
        acrossFilesystems: Bool = false,
        ceilingDirectories: [URL] = []
    ) throws(GitError) -> Repository
}
```

Implementation detail: `ceilingDirectories` is passed to libgit2 as a single colon-separated string (libgit2's expected format). `acrossFilesystems` maps to the `across_fs` parameter.

## 12. API Design Notes

- `object(for:)` and `reference(named:)` both return **Optional** ("not found" is Optional; "genuine failure" throws). The similar `commit(for:)` from v0.2 **throws** `.notFound` — that asymmetry is intentional: a caller who has a `Commit.oid` expects the commit to exist, while `object(for:)` / `reference(named:)` are exploratory. v0.3 does not revisit `commit(for:)`.
- `Tree` does not conform to `Sequence` / `Collection` in v0.3. Keeping it index-based matches the few places in `Git2` that return class handles with sequence-like payloads (e.g. the new `Diff`).
- `RevWalk` **is not** a `Sequence` (throws problem, see §8.1). `CommitSequence` **is** a `Sequence` (silent-fail tradeoff, see §8.3).
- `TreeEntry`, `DiffDelta`, `DiffFile` are all snapshot values, not borrowed views onto libgit2 memory. This is what lets them be `Sendable` / `Equatable` honestly.
- `Object` is an enum; `Commit` / `Tree` / `Blob` / `Tag` are classes. The enum case payload is a single reference, so `Object` stays cheap to pass around.

## 16. Implementation Order

Per the brainstorming conclusion, v0.3 follows Approach 1 (internal helpers first, then the public types that use them). Target order for the implementation plan:

1. `OID(hex:)` migration to `git_oid_fromstrn` (isolated; unblocks nothing but clears a deprecation).
2. `ObjectKind` + `ObjectHandle` internal helper.
3. `Tree` + `TreeEntry` + `TreeEntry.FileMode`.
4. `Blob`.
5. `Tag`.
6. `Object` enum + `Repository.object(for:)`.
7. Public `RevWalk` class + `CommitSequence.Sorting` + `log(from:sorting:)` overload. Internal `RevWalkHandle` deleted. `CommitSequence` / `CommitIterator` rewired.
8. `ReferenceSequence` + `Repository.references()` + `Repository.reference(named:)`.
9. `Repository.discover(...)` + `Repository.open(discoveringFrom:...)`.
10. `Diff` + `DiffDelta` + `DiffFile` + `Repository.diff(from:to:)`.
11. Annotated-tag peel test in `ReferenceTests` (the deferred v0.2 test).
12. Concurrency smoke test over all the new surfaces.

Each step is landable as its own commit with its own tests.
